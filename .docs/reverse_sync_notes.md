# Обратная синхронизация лайков (расширение → Boosty DOM) — Заметки разработки

**Дата:** 2026-06-11
**Задача:** При установке/снятии чекбокса в расширении, лайк должен визуально обновляться на странице Boosty в реальном времени (без перезагрузки F5).

---

## Текущее состояние проблемы

### Что работает ✅
- **Boosty → расширение:** При клике на лайк на Boosty, `page_script.js` (main world) перехватывает fetch/XHR запрос к `/reaction` и через `postMessage(LF_REACTION_INTERCEPTED)` мгновенно обновляет чекбокс в расширении.
- **API-запросы из расширения:** Функции `sendBoostyReaction(postId)` и `removeBoostyReaction(postId)` в `content.js` корректно отправляют POST/DELETE запросы на `api.boosty.to`. Лайк ставится/снимается на СЕРВЕРЕ Boosty (проверено — при F5 лайк есть).
- **Прямые вызовы API восстановлены:** В `content.js` убрана сложная логика `pendingDomClicks` с таймерами. API-запрос отправляется сразу же при клике на чекбокс, что гарантирует сохранение лайка на сервере.
- **Визуальное обновление DOM через React-клики:** После отправки API-запроса, `content.js` отправляет сообщение `LF_TOGGLE_LIKE_DOM` в `page_script.js`. Тот находит кнопку реакции поста, программно кликает по ней (открывая popover), дожидается появления иконки "heart" в popover (таймаут 1.5с) и кликает по ней.

### Что было решено (Текущее состояние) ✅
- **Баг со снятием лайка через чекбокс:** Ранее при попытке убрать галочку, React отправлял дублирующий сетевой запрос (POST или DELETE), который конфликтовал с запросом от `content.js` (либо откатывал интерфейс из-за ошибки 404, либо происходило "переключение" статуса дважды).
- **Блокировка эхо-запросов (Решение):** В `page_script.js` добавлен Set `simulatedPosts`. При программном клике из расширения, мы добавляем `postId` в этот Set. Перехватчики `fetch` и `XHR` проверяют этот Set и, если видят программный клик, **блокируют** отправку сетевого запроса React и возвращают ему фейковый ответ `200 OK` (`new Response('{}')`). Это позволяет React'у успешно обновить свой UI (optimistic update), не ломая при этом состояние сервера и не создавая гонку.
- **Унификация клика для снятия/постановки (Решение 2):** Поскольку API и интерфейс Boosty работают по принципу переключателя (toggle), и для снятия, и для установки реакции в `simulateReaction` используется единая последовательность действий: клик по кнопке реакции -> ожидание всплывающего popover -> клик по иконке сердечка в popover. Это устраняет ложные проверки состояния активности.
- **Логика API (Решение 3):** Установка реакции отправляется POST-запросом. Снятие реакции работает через отправку DELETE-запроса на тот же эндпоинт. Функция `removeBoostyReaction` в `content.js` отправляет `DELETE` запрос.

## Архитектура (два мира Chrome Extension)

```
┌─────────────────────────────┐   ┌──────────────────────────────┐
│    content.js               │   │     page_script.js           │
│    (isolated world)         │   │     (main world)             │
│                             │   │                              │
│  Имеет доступ к:            │   │  Имеет доступ к:             │
│  - chrome.storage           │   │  - window.fetch (нативный)   │
│  - chrome.runtime           │   │  - XMLHttpRequest (нативный) │
│  - DOM (но свой fetch)      │   │  - React internals           │
│  - localStorage (общий)     │   │  - DOM (общий)               │
│                             │   │                              │
│  НЕ может:                  │   │  НЕ может:                   │
│  - Перехватить fetch сайта  │   │  - Использовать chrome API   │
│  - Взаимодействовать с      │   │                              │
│    React-стейтом            │   │                              │
└─────────────┬───────────────┘   └──────────────┬───────────────┘
              │                                   │
              └──── window.postMessage ───────────┘
```

**ВАЖНО:** `fetch` из `content.js` (isolated world) НЕ перехватывается monkey-patch в `page_script.js` (main world). Это разные объекты `window.fetch`. Таким образом, реальные гарантированные запросы уходят из `content.js`, а `page_script.js` только манипулирует DOM и блокирует дублирующие запросы React.

## Резюме последних изменений в коде (Решение проблемы)

### В файле `content.js`
- В `removeBoostyReaction` используется метод `DELETE` для снятия лайка, что корректно обрабатывается API Boosty.

### В файле `page_script.js`
- Добавлен `const simulatedPosts = new Set()` для отслеживания "наших" кликов.
- В перехватчике `window.fetch` и `XMLHttpRequest.prototype.send` добавлена логика: если `postId` находится в `simulatedPosts`, оригинальный метод не вызывается, а возвращается Promise с фейковым успешным ответом.
- В `simulateReaction` убрано неработающее ветвление с `data-active` и осуществлен переход на единую схему: клик по кнопке -> ожидание popover -> клик по сердечку в popover (работает как для установки, так и для снятия лайка).
- Вокруг клика устанавливается блокирующий флаг в `simulatedPosts`, который снимается через `setTimeout(..., 500)` для уверенного перехвата.

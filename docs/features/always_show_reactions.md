# Функция «Всегда раскрывать лайки (меню реакций)»

Функция предназначена для того, чтобы меню выбора реакций (смайликов) под постами на Boosty оставалось раскрытым постоянно, без необходимости наводить курсор мыши на кнопку реакций.

---

## 1. Проблемы оригинального поведения Boosty
1. **Динамический рендеринг (Unmounting):** Меню реакций на Boosty является React-порталом (использует библиотеки позиционирования вроде Floating UI / Radix UI). При уходе курсора мыши React полностью удаляет (размонтирует) меню реакций из DOM.
2. **Ограничение на один открытый поповер (Синглтон):** Всплывающее меню реакций спроектировано как синглтон — на странице одновременно может быть открыто меню **только для одного поста**. Попытка эмулировать наведение курсора сразу на все посты страницы вызывает конфликт рендеринга в React (поповеры бесконечно закрывают друг друга, создавая эффект мерцания и высокую нагрузку на ЦП).

---

## 2. Архитектура технического решения

Для решения этих проблем были внедрены три взаимосвязанных механизма:

### А. Эксклюзивное удержание активного меню (Focusing viewport)
Чтобы избежать конфликтов синглтона и мерцания, расширение автоматически определяет **один активный пост**, который в данный момент читает пользователь (находится ближе всего к центру экрана по вертикали).
- Метод `autoOpenReactions()` находит все кнопки реакций в viewport, рассчитывает их расстояние до центра экрана и выбирает наиболее подходящую кнопку.
- Для этой активной кнопки эмулируется наведение (`mouseenter`), а для остальных кнопок посылается `mouseleave`, чтобы дать React плавно закрыть неактивные попапы.
- Периодичность обновления: каждые **1000 мс** через `setInterval`, а также мгновенно (с дебаунсом в **150 мс**) при прокрутке страницы (`scroll`).

### Б. Блокировка событий скрытия в фазе Capture
Когда курсор мыши уходит с кнопки реакций или из самого открытого меню, браузер генерирует события ухода мыши. Мы перехватываем их на уровне `window` в фазе **capture** (до того, как их получит React-приложение) и блокируем:
- Перехватываемые события: `mouseleave`, `mouseout`, `pointerleave`, `pointerout`.
- Если цель события (`e.target`) или элемент, на который переходит курсор (`e.relatedTarget`), является частью меню реакций или кнопкой реакций, распространение события останавливается через `e.stopImmediatePropagation()`.
- Также события блокируются, если курсор полностью уходит за пределы окна браузера или на второй монитор (`e.relatedTarget === null` или `e.toElement === null`).

### В. Геометрическая проверка состояния
Для предотвращения циклического вызова событий наведения расширение проверяет, открыто ли уже меню реакций для выбранной кнопки.
- **По ID:** Сначала проверяется стандартный атрибут `aria-describedby` на кнопке реакций или её дочерних элементах. Если элемент с таким ID существует в DOM, меню считается открытым.
- **По расстоянию (fallback):** Если связь по ID отсутствует, используется геометрическая проверка. Метод `isPopoverOpenForBtn` ищет элемент с классом попапа (`ReactionSelector`/`TooltipContent`) и проверяет его координаты относительно кнопки реакций. Если они расположены рядом по горизонтали (в пределах 150px) и вертикали (в пределах 120px), меню считается открытым.

### Г. Обработка кликов
Любой клик на странице (включая клик по смайлику для установки лайка или клик по боковой панели расширения) заставляет React принудительно закрыть попапы. Чтобы вернуть их обратно, глобальный обработчик кликов `handleGlobalClick` запускает автооткрытие `autoOpenReactions()` через **150 мс** после клика.

## 3. Точки интеграции в коде

Все изменения изолированы в блоках `/* DEV_ONLY_START */` ... `/* DEV_ONLY_END */`.

### В файле `content.js`
*   **`isReactionElement(el)`** — рекурсивно поднимается вверх по DOM-дереву без ограничения глубины, проверяя классы (`ReactionSelector`, `TooltipContent`, `ReactionsPost`, `ReactionButton`, `Reaction`) и атрибут `data-test-id="COMMON_REACTIONS_REACTIONSPOST:ROOT"`, чтобы определить принадлежность элемента к блоку реакций.
*   **`handleReactionLeave(e)`** — блокирует события закрытия.
*   **`handleGlobalClick(e)`** — запускает переоткрытие после кликов.
*   **`handleDevScroll()`** — троттлит запуск обновления при скролле (150 мс).
*   **`isPopoverOpenForBtn(btn)`** — проверяет статус видимости меню (через ID и геометрические координаты).
*   **`autoOpenReactions()`** — ключевой планировщик: находит центральный пост на экране, закрывает другие меню и открывает активное.
*   **`applyDevSettingsEffects()`** — инициализирует и удаляет слушатели событий (`mouseleave`, `mouseout`, `pointerleave`, `pointerout`, `click`, `scroll`) и интервал проверки при включении/выключении опции.
*   **`cleanup()`** — полностью очищает все слушатели при инвалидации контекста расширения.

### В файле `styles.css`
*   Отключаются CSS-переходы, анимации появления/исчезновения и принудительно задаются правила видимости для предотвращения мерцания:
```css
body.lf-dev-always-show-reactions [class*="ReactionSelector"],
body.lf-dev-always-show-reactions [class*="TooltipContent"] {
  transition: none !important;
  animation: none !important;
  pointer-events: auto !important;
  opacity: 1 !important;
  visibility: visible !important;
}
```

## 4. Возможные точки отказа (при обновлении разметки Boosty)

Если в будущем функция перестанет работать, проверьте следующие элементы:
1.  **Селектор кнопки реакций:** В `content.js` используется `[data-test-id="COMMON_REACTIONS_REACTIONSPOST:ROOT"]`. Если Boosty изменит этот `data-test-id`, расширение не сможет находить кнопки.
2.  **Классы поповера реакций:** В `content.js` и `styles.css` используются неполные селекторы классов по вхождению подстроки (`ReactionSelector`, `TooltipContent`, `ReactionsPost`, `ReactionButton`, `Reaction`). Если Boosty перейдет на обфусцированные классы без этих ключевых слов, логика перехвата перестанет работать.
3.  **Атрибут поповера:** Проверка роли `role="tooltip"` на поповере.

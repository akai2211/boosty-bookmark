# Технические детали реализации функций: Boosty Bookmark

В данном документе собраны подробности технической реализации дополнительных возможностей расширения, которые были интегрированы на этапе развития интерфейса и настроек. Это позволяет сохранить контекст разработки и упростить дальнейшую поддержку кода.

---

## 1. Вкладка «Настройки»
Вкладка интегрирована в боковую панель как отдельный экран, сменяющий основной список тайтлов при переключении состояния `state.ui.activeTab === 'settings'`.

*   **Логика переключения:** Управление отображением происходит в функции `render()` в `content.js`. При активации вкладки вызывается функция `renderSettingsContent()`, которая заменяет содержимое контейнера `#lf-body-content`.
*   **Сохранение настроек:** Все изменения параметров (`syncLikes`, `autoMarkOpen`, `zoom`, `tabOrder`) немедленно сохраняются в локальное хранилище Chrome (`chrome.storage.local`) через функцию `saveStateToStorage()` и вступают в силу без перезагрузки страницы.

---

## 2. Экспорт и импорт прогресса (ZIP-архив)
Реализован ручной механизм резервного копирования и переноса прогресса пользователя (статусы, заметки в блокноте, просмотренные посты, настройки).

*   **Экспорт (`exportUserData`):** 
    Запрашивает все данные из локального хранилища `chrome.storage.local` (для поддержки бэкапа сразу по всем каналам автора). Используя библиотеку `JSZip`, формирует структуру папок для каждого канала (название папки совпадает с `slug` канала, например `lightfoxmanga`). Внутри каждой папки создается файл `progress.json`, который содержит версию структуры данных, дату экспорта, кэш постов (`posts`), прогресс пользователя (`user_data`), настройки расширения (`settings`), свернутые группы и ссылки профиля. Генерирует и инициирует скачивание ZIP-архива `boosty_bookmark_backup_YYYY-MM-DD.zip`.
*   **Импорт (`importUserData`):** 
    Принимает ZIP-архив, считывает его как `ArrayBuffer` и распаковывает средствами библиотеки `JSZip`. Проходит по всем файлам архива вида `имя_канала/progress.json`, считывает их и производит базовую валидацию. Все импортированные данные каналов сохраняются в локальное хранилище расширения (`lf_state_<slug>`). Если в импортируемом бэкапе присутствуют данные текущего активного канала, они немедленно загружаются в оперативную память расширения (`state`), после чего интерфейс обновляется.

## 3. Динамический порядок вкладок
Порядок вкладок больше не зашит жестко в разметке и рендерится динамически на основе массива настроек.

*   **Хранение:** Порядок хранится в `state.settings.tabOrder` в виде массива ключей вкладок.
*   **Отрисовка:** Функция `render()` обходит массив `state.settings.tabOrder` и генерирует HTML-код кнопок вкладок с соответствующими именами из словаря `TAB_NAMES`.
*   **Управление кнопками:** Рядом с каждым элементом в настройках выводятся кнопки ▲ и ▼. Функция `moveTab(index, direction)` меняет элементы массива местами, проверяя границы (блокируя кнопки `disabled` для крайних элементов), сохраняет состояние и вызывает полный ререндер интерфейса.

## 4. Нативный Drag and Drop для сортировки вкладок
Для более удобной сортировки вкладок в настройках реализован механизм Drag and Drop с помощью стандартного HTML5 Drag and Drop API.

*   **Безопасность перетаскивания (Draggable):** 
    Чтобы избежать случайного перетаскивания кнопок или текста, свойство `draggable="true"` присваивается элементу списка `.lf-tab-order-item` динамически только при событии `mousedown` на ручке перетаскивания `.lf-drag-handle` (иконка с 12 точками). При событиях `mouseup` и `dragend` перетаскиваемость сбрасывается в `false`.
*   **События Drag & Drop:** 
    *   `dragstart` — запоминает индекс перетаскиваемой вкладки.
    *   `dragenter` / `dragleave` — добавляет/удаляет класс `.lf-drag-over` для подсветки целевого элемента.
    *   `dragover` — отменяет стандартное поведение браузера для разрешения сброса (`drop`).
    *   `drop` — вызывает функцию `dragAndDropReorder(fromIndex, toIndex)`.
*   **Перестроение массива:** 
    Функция `dragAndDropReorder` вырезает элемент из исходного индекса через `splice` и вставляет его на новый целевой индекс, после чего сохраняет настройки и обновляет UI.
*   **CSS оптимизация:** 
    Для дочерних элементов внутри перетаскиваемой строки в CSS прописано `pointer-events: none` (кроме ручки и кнопок), чтобы при перетаскивании браузер не генерировал ложные события ухода мыши (`dragleave`) с родительского контейнера.

## 5. Сворачиваемые секции в настройках
Для экономии пространства на вкладке настроек блоки «Синхронизация и бэкап» и «Порядок вкладок» сделаны сворачиваемыми (collapsible).

*   **Состояние видимости:** За развернутое состояние секций отвечают переменные `state.ui.syncBackupExpanded` и `state.ui.tabOrderExpanded` соответственно (хранятся во временном состоянии интерфейса, сбрасываясь при закрытии страницы).
*   **Рендеринг и CSS:** 
    *   При значении `true` к контейнерам соответствующих секций добавляется класс `.lf-expanded`.
    *   По клику на шапки секций (`#lf-toggle-sync-backup` и `#lf-toggle-tab-order`) состояние инвертируется и запускается `render()`.
    *   В `styles.css` тело секции `.lf-settings-section-body` по умолчанию скрыто (`display: none`) и раскрывается в `display: block` только при наличии класса `.lf-expanded` у родительской секции.
    *   Иконка стрелочки `.lf-collapse-arrow` плавно поворачивается на 180 градусов благодаря `transition: transform 0.2s`.

## 6. Масштабирование панели (Zoom)
Позволяет пользователям кастомизировать масштаб выдвижной панели под разрешение экрана.

*   **Настройка масштаба:** В секцию интерфейса добавлен селектор выбора масштаба (от 80% до 150%). Значение записывается в `state.settings.zoom` (по умолчанию в интерфейсе установлено значение 100%, что физически соответствует коэффициенту 1.25).
*   **Применение:** При создании панели в функции `createSidebar()`, а также при изменении селектора в реальном времени, к DOM-элементу панели `#lf-sidebar` применяется CSS-переменная `--lf-zoom`: `sidebar.style.setProperty('--lf-zoom', state.settings.zoom);`. В CSS-файле `styles.css` значение переменной считывается через `--lf-zoom-val` и применяется как `zoom: var(--lf-zoom-val);`, а высота сайдбара динамически рассчитывается как `height: calc((100vh - 56px) / var(--lf-zoom-val));` для предотвращения перекрытия хедера и корректной работы скроллбара.

## 7. Поведение свертывания подкатегорий (групп тайтлов)
Обеспечивает удобную навигацию по большому числу тайтлов на вкладках «Все» и «Новые» за счет их свертывания по умолчанию.

*   **Состояние свернутости по умолчанию:** Переменная `isCollapsed` в функции `renderGroup` вычисляется как `state.collapsedGroups[groupName] !== false`. Это означает, что если группа ранее не была явно развернута пользователем (её значение в объекте не равно `false`), она считается свернутой.
*   **Инверсия логики переключения:** При клике на заголовок группы, если она сворачивается, устанавливается `state.collapsedGroups[groupName] = true`, а если разворачивается — `state.collapsedGroups[groupName] = false`.
*   **Сброс при полной синхронизации:** Во время ручного запуска полной синхронизации базы постов (`performFullSync`) объект `state.collapsedGroups` принудительно очищается (`{}`), благодаря чему после обновления все категории автоматически возвращаются в исходное свернутое состояние.

## 8. Защита от ошибки «Extension context invalidated»
При обновлении или перезагрузке расширения в Chrome старый content-скрипт, уже внедрённый в страницу, продолжает выполняться. Однако его контекст (`chrome.runtime`, `chrome.storage`) становится недействительным, и любой вызов Chrome API выбрасывает ошибку `Uncaught Error: Extension context invalidated`. Это приводило к невозможности обновить расширение без его полного удаления.

*   **Функция `isExtensionContextValid()`:** Проверяет доступность контекста через `chrome.runtime && chrome.runtime.id`. Если `chrome.runtime.id` отсутствует или выбрасывает исключение, контекст считается мёртвым. Вызывается перед каждым обращением к Chrome API.
*   **Функция `cleanup()` и отписка от событий:** Останавливает периодический интервал проверки URL, удаляет все DOM-элементы расширения и **отписывается от всех глобальных слушателей событий** (`popstate`, `hashchange`, `click`, `beforeunload`), ссылки на которые хранятся в объекте `eventHandlers`. Это предотвращает срабатывание "мертвых" функций при навигации пользователя. Также восстанавливаются оригинальные методы `history.pushState` и `history.replaceState`.
*   **Безопасные обработчики:** Во все обработчики событий добавлена проверка `isExtensionContextValid()`. При попытке браузера вызвать обработчик из инвалидированного контекста, происходит вызов `cleanup()` и мгновенный возврат. Вызовы `window.dispatchEvent` дополнительно обернуты в `try/catch`.
*   **Обёртки Chrome API:** Все вызовы `chrome.storage.local.get/set` (в `loadStateFromStorage`, `saveStateToStorage`, обработчик `beforeunload`) обёрнуты в проверку контекста + `try/catch`, чтобы предотвратить необработанные исключения.
*   **Проверка `chrome.runtime.lastError`:** В колбеках `chrome.storage` дополнительно проверяется `chrome.runtime.lastError`, чтобы предотвратить ошибки, возникающие если контекст инвалидируется между вызовом и колбеком.

## 9. Синхронизация статуса лайков (isLiked)
Для того чтобы API возвращало актуальный статус пролайканных постов (поле `isLiked: true`), запросы к API (`https://api.boosty.to/...`) должны совершаться с использованием текущей сессии пользователя. 

*   **Авторизация через токен:** Одних только cookies для API Boosty (cross-origin запросы с `boosty.to` на `api.boosty.to`) оказалось недостаточно. API требует передачи токена доступа.
*   **Решение (getBoostyAuthToken):** В контент-скрипт добавлена функция `getBoostyAuthToken`, которая считывает токен авторизации пользователя (`accessToken`) из `window.localStorage` (ключ `auth`), где Boosty хранит активную сессию.
*   **Использование:** При каждом вызове `fetch` (как в `performFullSync`, так и в `backgroundSync`) этот токен подставляется в HTTP-заголовки запроса в виде `Authorization: Bearer <token>`. Это заставляет API Boosty распознавать пользователя и корректно отдавать статус `isLiked` для каждой главы произведений.

## 10. Стилизация чекбоксов просмотренных глав
Для улучшения видимости в темной и светлой темах, цвет чекбоксов изменен с системного серого на оранжевый (в стиле Boosty).

*   **Проблема с атрибутом `disabled`:** Ранее чекбоксы пролайканных на Boosty глав имели атрибут `disabled`, что заставляло браузер (особенно Chrome) принудительно окрашивать их в блеклые, нечитаемые цвета, игнорируя любые CSS-свойства вроде `accent-color`.
*   **Решение:** Атрибут `disabled` был удален. Вместо него пролайканные главы получают класс `.lf-liked-checkbox`.
*   **Окрашивание:** Для всех чекбоксов глав (`.lf-chapter-checkbox`) установлено свойство `accent-color: var(--lf-primary);`, что делает активные галочки оранжевыми во всех темах браузера.
*   **Снятие лайков:** Ранее чекбоксы пролайканных постов блокировались (через `pointer-events: none` и JS). Теперь эта блокировка снята: пользователь может свободно снимать галочки. При снятии галочки отправляется `DELETE` запрос на API Boosty (`removeBoostyReaction`), а кэш поста обновляется.

## 11. Сохранение состояния открытости панели и рендеринг при инициализации
Для устранения бага, когда после обновления страницы (F5) боковая панель могла отображаться пустой (только размытый фон backdrop), были внесены следующие изменения:

*   **Сохранение открытости (`sidebarOpen`):** 
    В настройки расширения (`state.settings.sidebarOpen`) добавлено логическое свойство, которое сохраняется в локальное хранилище Chrome при клике на кнопку-триггер, клике снаружи для закрытия или при автоматическом скрытии при уходе с целевой страницы.
*   **Восстановление состояния:** 
    При создании контейнера панели в `createSidebar()` проверяется сохраненное значение `state.settings.sidebarOpen`. Если оно равно `true`, к боковой панели сразу добавляется класс `.lf-open` для её плавного выдвижения.
*   **Принудительный рендеринг при старте:** 
    Ранее при наличии постов в кэше (`state.posts.length > 0`) контент-скрипт запускал фоновое обновление `backgroundSync()`, но не вызывал функцию `render()`. Из-за этого при перезагрузке страницы DOM-структура панели оставалась пустой, пока не приходил ответ от API Boosty с какими-либо обновлениями. Теперь при наличии постов функция `render()` вызывается незамедлительно при создании интерфейса, заполняя сайдбар сохраненным в памяти кэшем глав, и только затем запускается фоновый запрос обновлений.

## 12. Позиционирование панели и логика скрытия кнопки-триггера
Для того чтобы интерфейс расширения не перекрывал шапку (header) сайта Boosty, и для улучшения UX при открытии сайдбара, были внесены следующие изменения:

*   **Вертикальный сдвиг сайдбара с учетом масштабирования (zoom):**
    В CSS для контейнера `#lf-sidebar` заданы свойства `top: 56px;` (координаты fixed-позиционирования не масштабируются браузером при применении `zoom`, поэтому отступ сверху всегда составляет физические `56px`) и `height: calc((100vh - 56px) / var(--lf-zoom-val));` (деление на коэффициент масштабирования компенсирует растяжение высоты). Это гарантирует, что при любом масштабе сайдбар на экране располагается ровно под шапкой Boosty, не перекрывая элементы хедера, а его физическая высота остается равной доступной высоте вьюпорта, что сохраняет работоспособность внутреннего скроллбара.
*   **Скрытие плавающей кнопки-триггера:**
    В CSS добавлен декларативный селектор соседа `#lf-sidebar.lf-open ~ #lf-trigger-btn { display: none !important; }`. При открытии панели кнопка-триггер автоматически скрывается, разгружая интерфейс.
*   **Порядок создания элементов в DOM:**
    В функции `checkUrlAndToggleVisibility()` изменен порядок вызовов: теперь сначала вызывается `createSidebar()`, а затем `createTriggerButton()`. Это гарантирует, что кнопка-триггер идет после сайдбара в дереве DOM, что необходимо для корректной работы CSS-селектора общего соседа (`~`).
*   **Кнопка закрытия (сворачивания) в шапке панели:**
    В верстку шапки панели (справа) добавлена круглая кнопка сворачивания `#lf-close-btn` с SVG-иконкой стрелочки вправо. При нажатии на нее панель плавно уезжает вправо, а состояние `state.settings.sidebarOpen` сбрасывается в `false`. Автоматическое закрытие панели при клике снаружи (потеря фокуса) было полностью отключено по просьбе пользователя для повышения стабильности UX.

## 13. Автоматическая постановка лайка (реакции) при отметке просмотренного
Для обеспечения бесшовной интеграции с экосистемой Boosty, при ручной установке галочки чекбокса в расширении теперь автоматически отправляется запрос на постановку лайка к этому посту.

*   **Endpoint API:** Запрос отправляется методом `POST` на `https://api.boosty.to/v1/blog/lightfoxmanga/post/<postId>/reaction?from_page=blog`.
*   **Авторизация:** Используется тот же токен (`accessToken`), получаемый из `localStorage` браузера пользователя.
*   **Payload:** Отправляется в формате `application/x-www-form-urlencoded` с телом `reaction=heart`, что соответствует стандартному лайку (сердечку) на Boosty.
*   **Логика:** При установке галочки вызывается `sendBoostyReaction(postId)` (POST), при снятии — `removeBoostyReaction(postId)` (DELETE). Локальный кэш поста немедленно обновляется (`isLiked = true/false`), чтобы избежать рассинхронизации при последующем рендере.

## 14. Мгновенная синхронизация лайков через перехват fetch
Для обеспечения реактивности расширения без необходимости ручного обновления базы реализован перехват `window.fetch`. Когда пользователь ставит лайк на странице Boosty, расширение мгновенно обновляет свой кэш и интерфейс.

*   **Функция `patchFetch()`:**
    Сохраняет ссылку на оригинальный `window.fetch` в `eventHandlers.originalFetch` и подменяет его обёрткой-прокси. Обёртка прозрачно вызывает оригинальный fetch и параллельно анализирует URL и метод запроса.
*   **Условие перехвата:**
    Отлавливаются только `POST`-запросы, URL которых содержит `/reaction`. Из URL извлекается `postId` с помощью регулярного выражения `/\/post\/([a-f0-9-]+)\/reaction/i`. Это покрывает как запросы самого расширения (`sendBoostyReaction`), так и нативные запросы SPA-приложения Boosty при клике на кнопку лайка на сайте.
*   **Обработка результата (`handleInterceptedReaction`):**
    Вызывается асинхронно через `.then()` только при успешном ответе сервера (`response.ok`). Находит пост в кэше `state.posts` по `postId` и выставляет `isLiked = true`. Обновляет хранилище через `saveStateToStorage()`.
*   **Точечное обновление UI:**
    Если пользователь находится в детальном виде тайтла — ищет конкретный чекбокс по `data-post-id` и обновляет его состояние (`checked = true`, добавляет класс `lf-liked-checkbox`), а также пересчитывает счётчик прогресса в заголовке. Если пользователь в режиме списка — вызывает `renderListContent()` для обновления цветовых индикаторов.
*   **Восстановление в `cleanup()`:**
    При инвалидации контекста расширения оригинальный `window.fetch` восстанавливается из `eventHandlers.originalFetch`, чтобы не нарушать работу сайта Boosty после обновления расширения.

## 15. Двусторонняя синхронизация лайков в реальном времени (История решения)
**Задача:** Обеспечить мгновенное отображение изменений:
1. При постановке лайка на самом сайте Boosty галочка в расширении должна появляться без обновления страницы.
2. При постановке/снятии галочки в расширении, лайк на самом сайте Boosty должен появляться/исчезать без обновления страницы.

**Попытка 1: DOM-синхронизация (Программный клик - ОТКЛОНЕНО)**
Была попытка реализовать двустороннюю связь через поиск кнопки лайка в DOM и вызов `likeBtn.click()` из расширения.
*Что пошло не так:*
- `querySelector` цеплял ссылки из нашего же сгенерированного расширением сайдбара.
- Основная проблема: кнопка `[data-test-id="COMMON_REACTIONS_REACTIONSPOST:ROOT"]` на Boosty не отправляет запрос на лайк напрямую. Её нажатие лишь открывает React-меню (tooltip) выбора реакций. Из-за этого программный клик эмулировал открытие меню, но реального API-запроса на сервер Boosty не происходило.

**Попытка 2: Программный клик по поповеру реакций из `page_script.js` (ОТКЛОНЕНО, удалено 2026-06)**
Расширение через сообщение `LF_TOGGLE_LIKE_DOM` просило `page_script.js` симулировать клик по кнопке реакции + сердечку в поповере (`fireClick` с полной цепочкой pointer/mouse-событий) и сверяло счётчик «до/после» с retry. Разведка на текущей вёрстке Boosty (CDP, 2026-06) показала: поповер реакций — это **hover-tooltip**, и выбор реакции **не реагирует ни на синтетический, ни на trusted-клик**; счётчик на живой странице дрейфует (другие пользователи). Итог — ложный «рассинхрон счётчика» + спам в консоль. Весь блок симуляции (`simulateReaction`, `waitForReactionPopover`, `fireClick`, `getLikeCount`, `simulatedPosts`) удалён.

**Финальное решение: REST API (истина) + косметическая подсветка своим классом**
Подробности — в `docs/features/like_checkbox_sync.md`. Кратко:
*   **Направление «Лайк → Чекбокс»** (без изменений): `page_script.js` (MAIN world) перехватывает `window.fetch`/`XMLHttpRequest` POST/DELETE на `/reaction` и при успехе шлёт `LF_REACTION_INTERCEPTED` → `content.js` обновляет кэш и чекбоксы. Запросы самого расширения идут из isolated-world content-script и в этот перехватчик не попадают (эха нет).
*   **Направление «Чекбокс → Лайк»:** реальный лайк ставится **только** REST-запросом из `content.js` (`sendBoostyReaction`/`removeBoostyReaction`). Для мгновенной подсветки сердечка на странице `content.js` шлёт `LF_SET_LIKE_VISUAL`, а `page_script.js` вешает/снимает **свой** класс `lf-bb-liked` (стилизован внедрённым `<style>` — серая пилюля + оранжевый бейдж-счётчик). Свой класс не зависит от сгенерированного хэша Boosty `_selected_`, поэтому работает с первого раза на холодной странице.
*   **Проактивная синхронизация:** на `render()` (с дебаунсом) сайдбар шлёт `LF_SYNC_ALL_LIKES` со всеми лайкнутыми id; `page_script.js` подсвечивает их на странице. Плюс дебаунс-`MutationObserver` подсвечивает сердечко в лениво-отрисованном поповере.
*   **Поиск кнопки поста:** `findReactionButton` сначала проверяет, что URL страницы содержит `postId` (надёжно для страницы поста без комментариев), иначе ищет через `a[href*=postId]`.

## 16. Автоматический скролл к ленте постов
Для улучшения UX при переходе к конкретному тайтлу из карточки расширения реализован автоматический скролл мимо длинного блока «Об авторе» прямо к началу ленты постов (кнопкам «Лента» / «Медиа»).

*   **Триггер автоскролла (`checkAndScrollToFeed`):**
    Проверяет, содержит ли текущий URL параметры фильтрации Boosty (`postsTagsIds` или `tag`).
*   **Сравнение URL:**
    Чтобы предотвратить повторные нежелательные скроллы при прокрутке страницы пользователем, в замыкании сохраняется `lastScrolledUrl`. Скролл срабатывает только при переходе на новый уникальный URL.
*   **Ожидание элемента в DOM:**
    Поскольку Boosty работает как SPA, вкладки ленты могут отрисовываться с задержкой. Функция запускает интервал (до 3 секунд), опрашивающий DOM на наличие элемента `[class*="FeedTabs-scss--module_root_"]`.
*   **Смещение скролла:**
    Чтобы зафиксированная шапка Boosty (высотой 56px) не перекрывала кнопки «Лента / Медиа», прокрутка выполняется с вертикальным смещением в `-80px` от верхней границы элемента:
    `const y = rect.top + window.pageYOffset - 80; window.scrollTo({ top: y, behavior: 'smooth' });`
*   **Сброс состояния:**
    При вызове функции `cleanup()` переменная `lastScrolledUrl` сбрасывается в `null`.

## 18. Сохранение состояния интерфейса во вкладке через sessionStorage
Чтобы переходы в текущей вкладке (вызывающие полную перезагрузку страницы Boosty) не сбрасывали открытую карточку тайтла и текущую вкладку расширения, состояние UI временно кэшируется в `sessionStorage` браузера для этой вкладки.

*   **Сохранение состояния (`render`):**
    При каждом рендере интерфейса текущие значения `state.ui.activeTitle` и `state.ui.activeTab` записываются в `sessionStorage`:
    `sessionStorage.setItem('lf_active_title', state.ui.activeTitle || '');`
    `sessionStorage.setItem('lf_active_tab', state.ui.activeTab || 'favorite');`
*   **Восстановление состояния (`init`):**
    При инициализации расширения (после перезагрузки страницы или первоначальной загрузки) расширение считывает эти значения и восстанавливает их в `state.ui`. Если в `sessionStorage` был сохранен активный тайтл, расширение сразу рендерит детальный вид этого тайтла.
*   **Сброс при клике «Назад»:**
    Когда пользователь нажимает «Назад к списку», `activeTitle` сбрасывается в `null`, `render()` перезаписывает сохраненное значение пустой строкой, что гарантирует правильное поведение при последующих переходах.
*   **Очистка в `cleanup`:**
    При полной выгрузке расширения/инвалидации контекста ключи удаляются через `sessionStorage.removeItem`.

## 19. Бесшовная SPA-навигация при переходе по ссылке тега
Чтобы исключить полную перезагрузку страницы при клике на тег тайтла (которая вызывает мерцание, уничтожение DOM расширения и повторную его загрузку), в детальном виде реализован перехват события клика.

*   **Перехват клика (`renderDetailContent`):**
    На элемент `.lf-detail-title-link` вешается обработчик события `click`.
*   **Исключения (открытие в новой вкладке):**
    Если отключена настройка `openTitlesInCurrentTab` или пользователь зажал клавиши-модификаторы (Ctrl/Cmd) или нажал на колесико мыши (средний клик), обработчик позволяет выполнить стандартное поведение браузера (открытие в новой вкладке).
*   **Плавный SPA-переход:**
    В остальных случаях срабатывает `e.preventDefault()`, и переход осуществляется средствами React Router Boosty:
    1. Через `history.pushState({}, '', relativeUrl)` меняется адресная строка.
    2. Вызовом `window.dispatchEvent(new PopStateEvent('popstate', { state: {} }))` React Router сайта уведомляется о переходе, что заставляет его перерисовать ленту постов.
    3. Это вызывает нативное событие `lf_locationchange` расширения, и плавно скроллит страницу к началу ленты без перезагрузки всей страницы и закрытия сайдбара.

## 20. Сброс всех данных расширения
Позволяет полностью очистить локальное хранилище расширения и вернуть его к первоначальному состоянию.

*   **Интерфейс:**
    В самый низ вкладки настроек (под блоком «Инфо о расширении») добавлена скромная, невзрачная кнопка `#lf-delete-data-btn` (стилизована как второстепенная кнопка с блекло-красной границей и текстом, переходящими в насыщенный цвет при наведении).
*   **Подтверждение (Двухэтапное):**
    Во избежание блокировок браузера на вызовы нативных диалоговых окон (`confirm`) реализовано встроенное двухэтапное подтверждение:
    1. При первом клике кнопка меняет статус (`data-confirming="true"`), окрашивается в ярко-красный цвет, а её текст меняется на *«Вы точно уверены? Нажмите для удаления»*.
    2. Рядом динамически создается кнопка *«Отмена»* (при клике на неё кнопка возвращается в исходное состояние).
    3. Запускается таймер на 5 секунд: если повторный клик не совершен, кнопка автоматически сбрасывается в исходное состояние.
    4. При повторном клике по кнопке в активном состоянии подтверждения выполняется очистка данных.
*   **Логика очистки:**
    1. С помощью `chrome.storage.local.clear()` удаляются все данные из постоянного хранилища.
    2. Локальный объект состояния `state` сбрасывается к значениям по умолчанию.
    3. Очищаются временные данные в `sessionStorage` (`lf_active_title` и `lf_active_tab`).
    4. Масштаб боковой панели возвращается к дефолтному (100% / физический 1.25).
    5. Вызывается функция `render()`, которая перенаправляет пользователя на стартовый экран для выполнения первой синхронизации.

## 21. Переход и скролл к главе в ленте тайтла (Infinite Scroll)
Для повышения удобства чтения расширение позволяет переходить непосредственно к посту с выбранной главой в ленте тегов вместо открытия поста на отдельной странице.

*   **Настройка (`state.settings.openChaptersInFeed`):**
    По умолчанию **включена** (`true`). В настройки добавлен чекбокс «Переходить к главам в ленте тайтла» (`#lf-setting-open-chapters-in-feed`). В выключенном состоянии главы открываются как отдельные страницы постов (в текущей или новой вкладке, согласно настройке `openTitlesInCurrentTab`).
*   **Динамическая ссылка главы:**
    Если настройка включена, ссылка главы ведет на URL тега тайтла с хэшем `#post-[id]`. Если выключена — на прямую страницу поста `https://boosty.to/lightfoxmanga/posts/[id]`.
*   **SPA-переход для глав:**
    При переходе в текущей вкладке (`openTitlesInCurrentTab === true`) клик по ссылке главы всегда перехватывается: URL меняется через `history.pushState` (соответственно с хэшем или на прямую страницу поста), а событие `popstate` триггерит встроенный роутер Boosty для бесшовного SPA-перехода без перезагрузки страницы.
*   **Автоскролл с автоподгрузкой постов (`checkAndScrollToPost`):**
    Функция отслеживает появление хэша `#post-[id]` в URL (работает как при SPA-переходе, так и при открытии вкладки с нуля) и запускает интервал (`scrollToPostIntervalId`, шаг 600мс, до 50 попыток ≈ 30с; останавливается в `cleanup()`/`resetNavScrollState()` и при быстрой смене главы). Цикл двухфазный:
    *   **Фаза 1 — догрузка постов до появления цели.** Лента Boosty грузится не чистым infinite-scroll: первые порции по 5 постов подтягиваются автоскроллом, но после ~2 порций появляется кнопка **«Загрузить еще»**, которую нужно кликнуть, иначе подгрузка встаёт. Цикл на каждом тике: ищет целевой пост `findFeedPostElement()` (ссылку `a[href*="postId"]`, **исключая** панель расширения `#lf-sidebar`); если есть видимая кнопка «Загрузить еще» (`findFeedLoadMoreButton()`, RU/EN, без кнопок комментариев) — кликает её; иначе вызывает `nudgeFeedToLoadMore()`. Если прироста постов нет и кнопки нет ≥12 тиков — выходит (поста в ленте нет).
    *   **`nudgeFeedToLoadMore()` — почему не `scrollHeight`.** Скроллит к нижней границе **колонки постов**, а не к `document.scrollHeight`: правая колонка (тиры подписки) выше колонки постов, и прыжок в самый низ документа перелетает зону триггера ленивой подгрузки — посты не догружаются. Если позиция скролла не изменилась с прошлого тика («парковка»), сентинел уже сработал и не пере-сработает — делается «толчок» вверх, чтобы на следующем тике он снова вошёл в зону видимости.
    *   **Фаза 2 — приземление и стабилизация.** При нахождении цели берётся контейнер `.closest('[class*="Post-scss--module_root"]')`, первичный плавный скролл с отступом `-80px` под шапку Boosty, затем позиция удерживается мгновенными коррекциями, пока абсолютная координата поста не перестанет «ехать» (догрузка медиа выше может сдвигать цель) — фиксация после 2 стабильных тиков.
    *   **Проверено вживую (Playwright MCP, реальный Boosty):** тайтл «Пик боевых искусств» (40 постов) — переход к самому старому посту (низ списка, все 40 догружаются) и к среднему (стоп на 25): оба приземляются ровно под шапкой, без дрейфа.

## 22. Сортировка тайтлов на вкладках библиотеки
Для быстрого нахождения и упорядочивания произведений в боковой панели реализована интерактивная сортировка списка тайтлов.

*   **Интерфейс:**
    В строку поиска встроен flex-контейнер `.lf-search-row`. Справа от поля ввода размещена кнопка сортировки `#lf-sort-btn` с иконкой. При клике на неё раскрывается выпадающий список `#lf-sort-dropdown` с 8 режимами. Выбранный режим выделяется подсветкой `.lf-active`.
*   **Режимы сортировки (`state.settings.titleSort`):**
    *   `name_asc` / `name_desc` — По названию: А → Я / Я → А через сравнение `localeCompare`.
    *   `new_desc` / `new_asc` — По новизне (сначала новые / старые) на основе `publishTime` самого последнего поста в тайтле.
    *   `chapters_desc` / `chapters_asc` — По общему количеству глав (по убыванию / возрастанию).
    *   `progress_desc` / `progress_asc` — По прогрессу просмотра (сначала произведения с большим / меньшим количеством просмотренных глав, с дополнительной сортировкой по проценту просмотренного и алфавиту при совпадении).
*   **Сохранение и оптимизация:**
    Выбранный режим сортировки сохраняется в `state.settings.titleSort` и синхронизируется в `chrome.storage.local`. При изменении сортировки перерисовывается только список тайтлов (`renderListContent()`), что исключает потерю фокуса в поле поиска и обеспечивает моментальный отклик интерфейса.
*   **Исключение для вкладки «Новые»:**
    Вкладка «Новые» полностью игнорирует настройку `state.settings.titleSort`. Элементы в ней всегда принудительно сортируются по режиму `new_desc` (по новизне обновлений в порядке убывания), чтобы свежие обновления гарантированно выводились первыми и не терялись в списке.

## 23. Группа «Завершен том» во вкладке «Смотрю»
Для разгрузки вкладки «Смотрю» (`watching`) реализовано автоматическое отделение завершенных произведений:

*   **Критерий группировки:** Если у тайтла со статусом `watching` просмотрены все посты (`readCount === posts.length`) и этот тайтл помечен как завершенный том (`isVolumeFinished === true`), он считается завершенным томом.
*   **Отображение:** 
    *   Такие тайтлы автоматически убираются из общего списка отслеживаемых на вкладке «Смотрю» и опускаются в самый низ страницы.
    *   Они помещаются во встроенную группу «Завершен том», которая отрендерена с помощью стандартной функции `renderGroup()`.
    *   Группа является сворачиваемой (collapsible) и по умолчанию находится в свернутом состоянии. Её состояние свернутости сохраняется в `state.collapsedGroups['Завершен том']` и синхронизируется в локальном хранилище.

## 24. Архитектура и анализ WebDAV-синхронизации

В данном разделе приведён подробный анализ механизмов синхронизации в расширении и реализованная архитектура WebDAV-клиента в **Boosty Bookmark**.

### 24.1 Анализ ограничений chrome.storage.sync
Использование `chrome.storage.sync` накладывает жесткие лимиты со стороны Chrome API, что делает его непригодным для хранения больших объемов данных расширения:
- **QUOTA_BYTES** (Общий объем): **102 400 байт (100 КБ)**. База постов (`state.posts` для 1200+ постов) весит ~300 КБ. Синхронизировать её целиком в `sync` физически невозможно.
- **QUOTA_BYTES_PER_ITEM** (Размер ключа): **8 192 байта (8 КБ)**. Прогресс пользователя (`state.user_data`) хранится в виде одного объекта. При большом количестве тайтлов размер ключа превысит 8 КБ.
- **MAX_ITEMS** (Количество ключей): **512 ключей**.
- **Проблема кроссплатформенности:** `chrome.storage.sync` привязан к экосистеме конкретного браузера и не позволяет переносить данные между разными браузерами (например, Chrome и Firefox).

### 24.2 Архитектура WebDAV-провайдера (webdav-sync.js)
Для обеспечения свободы выбора пользователя и кроссплатформенности вся логика работы с облаком реализована в классе `WebDavProvider`. Синхронизация работает с любым WebDAV-сервером (Nextcloud, Owncloud, Яндекс.Диск, NAS и др.) с базовой авторизацией (Basic Auth) по логину и паролю приложения.

### 24.3 Формат файла синхронизации (boosty_bookmark_sync.zip)
Для экономии трафика и возможности многоканального бэкапа синхронизация осуществляется в виде ZIP-архива `boosty_bookmark_sync.zip`, который сохраняется в облаке в папке `boosty-bookmark/`. Формат архива полностью совместим с ручным импортом/экспортом: внутри находятся папки для каждого канала (имя совпадает со `slug`), а в них — файлы `progress.json`.

### 24.4 Слияние данных при синхронизации (Data Merging Strategy)
При выполнении синхронизации локальные данные из `chrome.storage.local` сливаются с удаленным архивом из облака с помощью функции `mergeChannelsMaps`:
1. **Слияние каналов:** Объединяются списки каналов из локальной базы и облака.
2. **Слияние постов (`posts`):** Посты объединяются по `id`. Статус лайка выставляется в `true`, если он был `true` хотя бы в одной версии (`isLiked: true`). `publishTime` берется максимальный.
3. **Настройки (`settings`):** Сравнивается дата экспорта (`exportDate`/`updatedAt`). Берутся настройки из более свежего бэкапа.
4. **Прогресс (`user_data`):** Слияние происходит для каждого тайтла индивидуально:
   - Статус (`status`) и текстовая заметка (`notes`) перезаписываются значениями из версии с более свежим `updatedAt`.
   - Массивы просмотренных глав (`readPosts`) объединяются (Union via Set).
5. **Таймстампы плееров (`playerTimestamps`):** Для каждого видеофайла сохраняется максимальный таймстамп (последний момент просмотра).
6. **Остальные данные:** Списки свернутых групп (`collapsedGroups`), ссылки (`blogDescriptionLinks`), списки новых тайтлов/глав объединяются.

## 25. Принудительное изменение качества видео (Force Video Quality) в плеере VK

В данном разделе описан механизм автоматической установки качества видео в веб-компоненте `<vk-video-player>`. Вся внутренняя структура плеера изолирована в **Shadow DOM** этого элемента.

### 25.1 Структура DOM плеера
1. Веб-компонент: `<vk-video-player>`
2. Контейнер со скрытым DOM: `div.shadow-root-container` (содержит `#shadow-root`)
3. Внутри Shadow Root:
   - Главная обертка плеера: `div.player-wrapper`
   - Контейнер интерактива: `div.container` (внутри `div.player-wrapper`)
   - Элемент видео: `<video>`
   - Меню настроек качества (появляется в разметке управления):
     - Элементы списка: `li.item-quality` (содержат `data-value` со значением разрешения, например `"1080p"`).

### 25.2 Алгоритм работы функции
Процесс принудительной установки качества разделен на три шага:
1. **Обнаружение плеера и доступ к Shadow DOM:** Скрипт находит контейнер `.shadow-root-container` и обращается к свойству `.shadowRoot`.
2. **Привязка к первому клику:** Поскольку элементы управления генерируются динамически при запуске, скрипт вешает однократный обработчик клика (`{ once: true }`) на элемент `div.player-wrapper div.container`. Это позволяет применить качество в момент старта видео.
3. **Программный клик по пункту качества:**
   - Производится поиск всех элементов `li.item-quality`.
   - Если найдено совпадение с выбранным пользователем качеством (например, `1080p`), выполняется `qualityEl.click()`.
   - Если точного совпадения нет, выбирается первый доступный вариант (обычно максимальный).

*(Примечание: в текущей версии расширения данная функция временно отключена из-за перехода плеера на двухуровневое меню и ленивую прорисовку элементов управления).*

## 27. Портирование и кросс-платформенная сборка для Firefox

С целью обеспечения кроссплатформенности расширение адаптировано для работы в браузере Firefox с сохранением совместимости сборки с Chrome/Chromium.

### 27.1 Конфигурация манифеста (manifest.firefox.json)
Для Firefox создан отдельный файл конфигурации `manifest.firefox.json` со следующими особенностями:
* **Фоновые скрипты (Background Scripts):** Вместо Chrome-специфичного параметра `"background": { "service_worker": "background.js" }` используется кроссплатформенный `"background": { "scripts": ["background.js"] }`. Это решает проблему несовместимости фоновых сервис-воркеров в Firefox MV3.
* **Идентификатор Gecko (Gecko ID):** В манифест добавлен обязательный для дистрибуции в экосистеме Mozilla блок `"browser_specific_settings": { "gecko": { "id": "boosty-bookmark@akai.io", "strict_min_version": "111.0" } }`. Минимальная версия установлена на `111.0`, так как именно с этой версии в Firefox гарантирована полноценная поддержка MAIN-окружения (`world: "MAIN"`) в контентных скриптах, которая необходима для перехвата лайков скриптом `page_script.js`.

### 27.2 Адаптация скрипта сборки (build.js)
Скрипт сборки `build.js` модифицирован для поддержки сборки под обе платформы:
* **Флаг сборки:** Добавлено считывание аргумента командной строки `process.argv.includes('--firefox')`.
* **Динамическая замена манифеста:** При сборке под Firefox файл `manifest.firefox.json` копируется во временную папку сборки и переименовывается в `manifest.json`. При сборке под Chrome используется стандартный `manifest.json`.
* **Специфичное имя архива:** В зависимости от переданного флага генерируется либо `boosty-bookmark-release.zip` (Chrome), либо `boosty-bookmark-firefox-release.zip` (Firefox).
* **Скрипт npm:** В `package.json` добавлена команда `npm run build:firefox` для сборки релиза под Firefox.

## 28. Порядок сортировки глав по умолчанию

Для обеспечения лучшего пользовательского опыта и соответствия стандартному поведению ленты постов Boosty, порядок отображения глав в детальном виде тайтла настроен следующим образом:
* **Поведение по умолчанию:** По умолчанию активирована сортировка «Новые вверху» (`state.ui.sortAsc: false`). Это значит, что самые свежие публикации/главы отображаются на первой позиции вверху списка, а более старые опускаются вниз.
* **Переключение порядка:** Пользователь может мгновенно изменить порядок сортировки с помощью интерактивной кнопки во главе списка («Новые вверху» / «Старые вверху»). При клике на неё значение `state.ui.sortAsc` инвертируется, а список глав перерисовывается.
* **Сортировка Объявлений:** Для тайтла «Объявления» сортировка всегда принудительно фиксируется в режиме «Новые вверху» (`state.ui.sortAsc = false`), так как данная вкладка отображает новостные посты автора.

## 29. Автоматический переход к тайтлу при выборе тега на Boosty

Для улучшения интеграции расширения с основным сайтом Boosty реализовано автоматическое открытие детального вида тайтла в сайдбаре при выборе тега (фильтрации постов) на странице Boosty:
* **Считывание параметров:** При каждом изменении URL-адреса страницы (перехват через события `lf_locationchange` и фоновый 500мс-интервал) вызывается функция `syncActiveTitleFromUrl()`. Она парсит URL-параметры `postsTagsIds` (ID тега на Boosty) или `tag` (название тега).
* **Сопоставление с тайтлами:** 
  1. Попытка сопоставить значение параметра с ID тега (`tagId`) сгруппированных тайтлов из `getGroupedTitles()`.
  2. Если сопоставление по ID не дало результатов (например, если ID не был сохранен), происходит поиск по имени тайтла, декодированному из URL-параметра (регистронезависимо).
* **Смена состояния сайдбара:** При нахождении совпадения `state.ui.activeTitle` автоматически переключается на имя найденного тайтла, состояние сохраняется в `sessionStorage` (для выживания при перезагрузке страницы), а интерфейс перерисовывается.
* **Защита от зацикливания:** Введена переменная `lastProcessedTagParam`, хранящая последний обработанный параметр тега. Это предотвращает повторное открытие детального вида по таймеру, если пользователь вручную нажал кнопку «Назад к списку» в сайдбаре при неизменном URL страницы.
* **Сброс при возврате:** Если пользователь переходит с теговой страницы обратно на общую ленту (параметр тега в URL исчезает), расширение сбрасывает `state.ui.activeTitle` в `null`, возвращая сайдбар к списку тайтлов.
* **Управление в настройках:** В раздел «Настройки отслеживания» добавлен параметр `syncTitleFromUrl` («Переход по тегам на Boosty»), который позволяет полностью включить или выключить автоматическую навигацию по тегам при клике на самом сайте. По умолчанию опция включена. Локализация реализована для русского и английского языков.

## 30. Модульная архитектура и сборка через esbuild

> Рефакторинг монолита `content.js` **завершён** (все этапы 1a–11, слиты в `main`; исторический план — в `docs/archive/split_plan.md`). Источник правды — ES-модули в `src/`; корневой `content.js` собирается esbuild. Раздел описывает итоговую архитектуру.

### 30.1 Источник правды — `src/`, `content.js` — артефакт
Исходный код переезжает в папку `src/` (ES-модули). Корневой `content.js` теперь **результат бандла** esbuild и **не коммитится** (внесён в `.gitignore`: `/content.js`, `/content.js.map`; убран из индекса через `git rm --cached`). Свежий клон/worktree собирает бандл командой `npm run build:js`. Перед загрузкой расширения в браузер обязательно держать запущенным `npm run watch:js`, иначе грузится устаревший бандл.

### 30.2 Бандлинг и манифесты
`locales.js` и `webdav-sync.js` переехали в `src/` как ES-модули с именованными `export` и **бандлятся внутрь** `content.js` (импортируются из `src/content.js`). Поэтому из секции `content_scripts → js` обоих манифестов (`manifest.json` и `manifest.firefox.json`) они удалены — остаются только `jszip.min.js` и `content.js`. `jszip.min.js` остаётся внешним глобалом (используется как `new JSZip()`), не бандлится. `page_script.js` (MAIN-world) собирается отдельно и копируется как есть.

### 30.3 Флаг `DEV` вместо маркеров `DEV_ONLY` (в JS)
Прежний механизм вырезания dev-кода через комментарии-маркеры `/* DEV_ONLY_START/END */` + regex несовместим с esbuild (он удаляет комментарии даже без minify). Поэтому **в JS** dev-код гейтится через глобальный флаг `DEV`, подставляемый esbuild на этапе сборки (`--define`):
* **dev-сборка** (`build:js`/`watch:js`): `--define:DEV=true` — весь dev-код сохраняется;
* **релиз** (`build.cjs`): `--define:DEV=false --minify-syntax` — esbuild сворачивает `if (false)` и tree-shaking'ом удаляет ставшие неиспользуемыми функции DevTools.
> `--minify-syntax` обязателен (иначе ветка `if (false) {}` остаётся). Важно: esbuild **не** инлайнит `const` в тернарное выражение — для гарантированного вырезания используется конструкция `if (DEV) { … }`, а не тернар. Маркеры `DEV_ONLY` остались только в `styles.css` (для CSS regex-механизм в `build.cjs` сохранён).

В юнит-тестах (Vitest импортирует `src/*.js` напрямую, минуя esbuild) флаг задан в `vitest.config.js`: `define: { DEV: 'false' }`, иначе голый `DEV` падает с `ReferenceError`.

### 30.4 Скрипт сборки `build.cjs`
`build.js` переименован в `build.cjs` (расширение `.cjs` нужно из-за `"type": "module"` в `package.json` — иначе CommonJS-скрипт ломается). Релизный `content.js` собирается через `esbuild.buildSync({ entryPoints: ['src/content.js'], bundle: true, define: { DEV: 'false' }, minifySyntax: true, charset: 'utf8' })` прямо в `.tmp_build/`. `charset: 'utf8'` сохраняет кириллицу читаемой (без экранирования в `\u`). `cleanDevCode` теперь применяется только к `.css`.

### 30.5 Тесты на ESM
`package.json` получил `"type": "module"`; тесты переписаны с CommonJS (`require`) на ESM (`import`):
* `tests/unit.test.js` грузит `src/content.js` через **динамический** `await import()` — это важно, так как статические `import` хойстятся выше установки моков `global.chrome`/`sessionStorage`, и `init()` отработал бы без моков;
* `manifest.json` подключается через `import manifest from '../manifest.json'` (Vitest резолвит JSON нативно);
* в E2E (`tests/extension.spec.js`) `__dirname` восстановлен из `import.meta.url` (в ESM он недоступен);
* добавлен npm-хук `"pretest": "npm run build:js"` — гарантирует свежий бандл перед `npm test` (E2E грузит `content.js` из корня).

| `src/ui/templates.js` | Чистые HTML-шаблоны интерфейса без побочных эффектов. |
| `src/ui/sidebar.js` | Рендеринг боковой панели: каркас, списки тайтлов, детальный вид, настройки, «О расширении», уведомления, drag&drop сортировка. |
| `src/ui/devtools.js` | Панель разработчика (**DEV-only**). Целиком вырезается из релиза tree-shaking'ом (все точки входа под `if (DEV)`). |

### 30.7 Разрыв циклов через внедрение зависимостей (`setXxxDeps`)
Граф импортов однонаправленный: `utils ← locales`; `state ← utils`; `grouping ← state, utils`; `sync ← state, utils, grouping`; `navigation ← utils, state, grouping`; `players ← state, utils`; `ui/sidebar ← state, utils, sync, grouping, templates`; `ui/devtools ← utils, state, sync, ui/sidebar`. Эти импорты делаются **напрямую**.

Там, где прямой импорт создал бы цикл (например, `sync` нужен `render` из `ui/sidebar`, а `ui/sidebar` импортирует `sync`), модуль объявляет **перепривязываемые dep-локали** (`let render = () => {}`) и экспортирует setter `setXxxDeps(d)`. `content.js` в начале `init()` вызывает `setStateDeps`/`setSyncDeps`/`setSidebarDeps`/`setNavigationDeps`, подставляя реальные реализации. Так циклы разорваны, а тела функций при переезде копировались дословно.

**Dev-зависимости** (`devSettings`, `applyDevSettingsEffects` из `ui/devtools.js`) проводятся в `sync`/`sidebar` **только под `if (DEV)`** — иначе ссылка на них удержала бы `ui/devtools.js` в релизной сборке и сломала tree-shaking (см. §30.3).

## 31. Логика вкладки «Новые» и время последнего визита (`lastVisit`)
* Категории вкладки — «Новые тайтлы» и «Новые главы»; группы всегда принудительно раскрыты (`isCollapsed = false`).
* Вкладка **не** подчиняется глобальной настройке сортировки: элементы всегда сортируются по дате последнего поста/обновления по убыванию (сначала новые), чтобы свежие обновления не терялись.
* После любой синхронизации (инкрементальной/полной/фоновой), если найдены свежие обновления, активная вкладка автоматически переключается на «Новые» (`state.ui.activeTab = 'new'`).
* `state.lastVisit` обновляется в хранилище **только при подтверждённом просмотре новинок**: при переключении с «Новые» на другую вкладку, сворачивании сайдбара кнопкой/триггером или по `beforeunload` — но лишь если сайдбар был открыт и активной была вкладка «Новые». Это защищает новинки от исчезновения, если пользователь обновил страницу, не открыв сайдбар.
* DevTools-эмуляция даты: при переходе во «вчера/прошлое» `lastVisit` инициализируется как `cutoffTimeMs − 24ч` (или при значении `0`), чтобы последние посты считались новыми; при сдвиге даты отсечки вперёд `lastVisit` сохраняется (не сбрасывается в `0`). Сохранение настроек DevTools запускает инкрементальную синхронизацию для мгновенного обновления UI.

## 32. Категории на вкладке «Все»: «Все», «Бесплатные», «Объявления»
* **Подкатегория «Все»:** на самой верхней позиции вкладки «Все» — плоский отсортированный список всех тайтлов без разбивки по уровням подписки (упрощает глобальный поиск и сортировку).

* **Снятие лимита `max-height`:** у `.lf-group-list` в `styles.css` лимит увеличен с `1000px` до `99999px` — иначе списки длиннее ~27 элементов обрезались свойством `overflow: hidden`.

## 35. WebDAV: клиентские детали (провайдер, логин, прогресс, ошибки)
* **Выбор провайдера** (`webdavConfig.provider`): `'yandex'` (по умолчанию) и `'webdav'`. Для Яндекса адрес сервера фиксируется на `https://webdav.yandex.ru`, поле адреса скрыто; для прочего WebDAV показываются поле адреса и инструкция Nextcloud. Динамические тексты/плейсхолдеры/уведомления зависят от провайдера. Автоопределение для старого конфига без поля `provider`: если `baseUrl` содержит `yandex` → `'yandex'`, иначе `'webdav'`.
* **Логин и код доступа:** учётные данные в `lf_webdav_config` (`baseUrl`, `username`, `accessCode`). В UI — термин «код доступа» (app password/token, не пароль аккаунта). Поле кода — `type="text"` + CSS `-webkit-text-security: disc` (против навязчивого автозаполнения) и кнопка-глазик (`state.ui.showAccessCode`). В поле имени пользователя кириллица удаляется регуляркой на `input`. При `provider === 'yandex'` почтовый домен Яндекса (`@yandex.ru`, `@ya.ru` и т.п.) автоматически отрезается в `createWebDavProvider` (визуально в поле остаётся как введено; корпоративные адреса не затрагиваются).
* **Прогресс и ошибки:** оранжевый прогресс-бар под кнопкой «Синхронизировать» (анимация до 95% за ~2.5 c). Флаг `state.ui.webdavSyncing` сбрасывается в блоке `finally` функции `performWebDavSync` вместе с `renderSettingsContent()`/`render()` — кнопка не зависает при любом исходе. При статусах `401`/`403` в методах `download`/`upload` выбрасывается понятная ошибка «Неверное имя пользователя или код доступа».
* **Автосинхрон и авто-выгрузка:** `triggerAutoWebDavSync()` при открытии сайдбара, не чаще 1 раза в минуту (`WEBDAV_AUTO_SYNC_MIN_INTERVAL_MS = 60000`); ручная кнопка — без лимита. После каждого успешного `saveStateToStorage()` запускается 20-секундный `debouncedWebDavUpload()` (повторные сохранения сбрасывают таймер, по истечении — тихая синхронизация).

## 36. Локализация: определение языка
Язык интерфейса определяет `getCurrentLang()`: сначала читает куку `locale` (`ru`/`en`) из `document.cookie` (доступна сразу при парсинге DOM), при отсутствии — фолбэк на `document.documentElement.lang`. Это устраняет race condition при перезагрузке по F5, когда React-код Boosty ещё не успел проставить атрибут `lang` у `<html>`. Функция `t(key, ...args)` подставляет плейсхолдеры `{0}`, `{1}`…; `tCategory(catName)` переводит названия тиров подписок, возвращая исходное название при отсутствии ключа перевода.

## 37. Устойчивость синхронизации: ретраи и докачка (`src/sync.js`)

Полная синхронизация (`performFullSync`) листает API Boosty постранично (по 100 постов). Раньше единичный сбой `fetch` на любой странице ронял весь цикл, и **все уже собранные посты выбрасывались** (`state.posts` присваивался только после полного успешного прохода). На блогах с 900+ постами это приводило к стабильному обрыву из-за rate-limit Boosty (после серии быстрых запросов API отдаёт ошибочный ответ без CORS-заголовков → `fetch` реджектится). Решение состоит из двух механизмов.

### 37.1 Ретрай с экспоненциальным backoff (`fetchPostsPage`)
Запрос страницы вынесен в хелпер `fetchPostsPage(offset, limit, maxAttempts = 4)`:
* при сетевом сбое / CORS-rejection / статусах `429` и `5xx` — ждёт и повторяет (паузы `1с → 2с → 4с`; для `429` учитывается заголовок `Retry-After`);
* клиентские ошибки `4xx` кроме `429` помечаются `nonRetryable` и пробрасываются сразу (повтор бессмыслен);
* после исчерпания попыток пробрасывает последнюю ошибку.

Хелпер используется в полной и инкрементальной синхронизации. Фоновая (`backgroundSync`) осталась одно-попыточной (best-effort, выполняется при каждом открытии страницы — ретраить нежелательно).

### 37.2 Докачка с чекпоинта
`performFullSync` после каждой успешно скачанной страницы сохраняет чекпоинт `{ offset, posts }` в **отдельный** ключ `chrome.storage.local` — `lf_full_sync_resume_<slug>`. Ключ намеренно вне пространства `lf_state_*`, поэтому **не попадает в ручной бэкап и WebDAV** (они фильтруют ключи по префиксу `lf_state_`).
* В начале синхронизации чекпоинт загружается: если он есть — `allPosts`/`offset` инициализируются им, и докачка продолжается **с места обрыва**, а не с нуля.
* На успешном завершении посты дедуплицируются по `id` (на случай перекрытия при докачке), сортируются «новые сверху», пишутся в `state.posts`, и чекпоинт **очищается**.
* При обрыве чекпоинт остаётся — следующий запуск («Попробуйте ещё раз») продолжит докачку. Прежняя база при неуспехе не затирается (`state.posts` меняется только после полного прохода).

Маппинг поста API → компактный формат вынесен в общий `toCompactPost(p)` (используется в полной/инкрементальной/фоновой синхронизации).

## 38. Группа «Просмотрены все главы» и динамический сдвиг сайдбара

Для улучшения пользовательского опыта (UX) и избежания наложения элементов управления на боковую панель реализованы две независимые фичи:

### 38.1 Группа «Просмотрены все главы»
В целях освобождения полезного пространства на вкладке «Смотрю» (`watching`) для отслеживаемых тайтлов, по аналогии с группой «Завершен том», внедрена группа «Просмотрены все главы» (`group_all_viewed`):
* **Критерий группировки:** Если у тайтла со статусом `watching` количество просмотренных глав равно общему числу глав (`readCount === posts.length` при `posts.length > 0`), но при этом флаг завершенности тома отсутствует (`isVolumeFinished === false`), тайтл считается полностью просмотренным (на текущий момент).
* **Поведение:** При включенной настройке `groupAllViewed` (по умолчанию `true`) такие тайтлы автоматически скрываются из основного списка и выносятся в свернутую по умолчанию группу «Просмотрены все главы» внизу вкладки.
* **Интерфейс:** В настройки сайдбара добавлен чекбокс «Выносить тайтлы с 100% прогрессом в группу «Просмотрены все главы»» с ключами локализации `settings_group_all_viewed_label` и `settings_group_all_viewed_desc`. При изменении состояния значение немедленно записывается в хранилище, а список перерисовывается.

### 38.2 Динамическое позиционирование сайдбара и ResizeObserver
Ранее верхний отступ (`top`) и общая высота (`height`) боковой панели расширения и dev-панели были жестко зафиксированы в `styles.css` на значениях `56px` и `calc(100vh - 56px)` соответственно, исходя из стандартной высоты хедера Boosty. Это приводило к наложению сайдбара на хедер или возникновению щелей при динамическом изменении высоты меню сайта (например, при адаптивной верстке или изменении масштаба браузера).
* **Использование CSS-переменной:** Свойства `top` и `height` сайдбара и dev-панели переведены на использование CSS-переменной `--lf-header-h` (по умолчанию `70px`), деленной на масштаб панели `zoom`:
  `top: calc(var(--lf-header-h, 70px) / var(--lf-zoom-val));`
  `height: calc((100vh - var(--lf-header-h, 70px)) / var(--lf-zoom-val));`
* **ResizeObserver:** В модуле `src/ui/sidebar.js` при рендеринге сайдбара инициализируется `ResizeObserver`, который следит за элементом хедера Boosty (`#TopMenu`). При изменении его размеров обсервер вызывает функцию `updateHeaderTop()`, вычисляющую актуальную позицию нижней границы хедера через `getBoundingClientRect().bottom` (по умолчанию `70px`) и динамически обновляющую значение `--lf-header-h` на корневом элементе `document.documentElement.style.setProperty`.
* **Очистка ресурсов:** При выгрузке расширения/сбросе контекста обсервер отключается с помощью вызова функции `cleanupHeaderObserver()`, что предотвращает утечки памяти.

## 39. Принудительное качество VK-плеера (Force Video Quality)
Видео на Boosty проигрываются веб-компонентом `vk-video-player` (новый VK-плеер с Shadow DOM). Прежняя реализация выставляла качество кликами по меню настроек плеера; после смены структуры плеера этот подход **нереализуем**: кнопку настроек (`button[data-testid="settings-btn"]`) нельзя открыть программно — она реагирует только на доверенный (`isTrusted`) ввод, а content-script такие события синтезировать не может. Пункты подменю качества (`li.item[data-value="1080p"|...]`) на синтетический клик реагируют, но без открытого меню их нет в DOM.

* **Рабочий механизм — подмена localStorage-ключа.** VK-плеер при инициализации читает `localStorage['vk_player_preferred_quality']` = `{"videoId":"<id>","value":"720p"}` и применяет качество, если `videoId` совпадает с текущим видео (при несовпадении — стирает запись и стартует в «Авто»). `videoId` доступен из публичного свойства Svelte-компонента `vk-video-player.store.videoId` ровно в момент чтения ключа (проверено: к этому моменту элемент уже в DOM и `store.videoId` установлен).
* **Перехват в `page_script.js` (MAIN world).** Переопределён `Storage.prototype.getItem`: при запросе ключа `vk_player_preferred_quality`, если функция включена и качество не «auto», возвращается `JSON.stringify({videoId: <из активного vk-video-player>, value: <целевое>})`. Так плеер получает корректный videoId на лету, без хрупких кликов и без записи в реальное хранилище. Подбор плеера — `getActiveVkVideoId()` (перебор `vk-video-player`, чей `store.videoId`/`videoConfig.videos[0].unitedVideoId` доступен).
* **Передача настройки.** Content-script шлёт в page_script `window.postMessage({type:'LF_SET_QUALITY_PREF', enabled, value})`: при инициализации (`src/content.js`, сразу после загрузки настроек — чтобы успеть до плеера) и при изменении настройки в сайдбаре (`src/ui/sidebar.js`). Хелпер `sendVideoQualityPref()` — в `src/players.js`.
* **Область применения.** Качество применяется к видео, открываемым **после** установки настройки (плеер читает ключ только при инициализации/перезагрузке); уже играющее видео не переключается на лету. Настройка (`forceVideoQuality`, `videoQuality`) разморожена в UI настроек; значения совпадают с `data-value` плеера (`144p`…`1080p`).

## 40. Автопубликация релиза на GitHub по тегу (`.github/workflows/release.yml`)
GitHub Release собирается и публикуется автоматически — **по пушу тега вида `vX.Y.Z`**, а не по обычному коммиту/пушу. Это сделано осознанно: можно пушить в `main` сколько угодно, релиз создаётся только когда выпуск намечен явным тегом.

* **Триггер.** `on: push: tags: ['v*']`. Обычные пуши в `main` workflow не запускают.
* **Шаги job'а (`ubuntu-latest`):** checkout → `actions/setup-node@v4` (Node 20, кэш npm) → `npm ci` → **проверка соответствия тега и версии** → `npm run test:unit` → `npm run build` + `npm run build:firefox` → публикация через `softprops/action-gh-release@v2`.
* **Защита от рассинхрона версий.** Шаг сверяет `${GITHUB_REF_NAME#v}` с `require('./manifest.json').version`; при несовпадении job падает с понятной ошибкой и релиз не создаётся (чтобы не выпустить архив со старой версией внутри). Поэтому порядок выпуска: сначала поднять версию в `manifest.json` (повышает **только Акай**) и запушить, потом ставить тег.
* **Артефакты релиза.** К Release прикрепляются оба архива (`build.cjs` именует их по версии из манифеста): `boosty-bookmark-v<ver>-release.zip` (Chrome) и `boosty-bookmark-v<ver>-firefox-release.zip` (Firefox). Архивы в `.gitignore` — генерируются в CI, в репозитории не лежат.
* **Права.** `permissions: contents: write` + встроенный `GITHUB_TOKEN` (локальный `gh auth login` не нужен). Если первый прогон падает с 403 — проверить **Settings → Actions → General → Workflow permissions** = «Read and write».
* **Описание релиза (release notes).** В workflow стоит `generate_release_notes: true` — он даёт лишь авто-«Full Changelog» со ссылкой на коммиты. Для каждого релиза нужно **человекочитаемое описание изменений** (что нового/исправлено, на русском, по пунктам). Источник пунктов — `docs/memory.md` и коммиты с прошлого тега; подавать через подробную аннотацию тега (`git tag -a`) с проброской в `body`, через `CHANGELOG.md` или правкой тела релиза после публикации.
* **Как выпустить релиз:**
  ```bash
  # (если нужна новая версия — сначала поднять её в manifest.json, закоммитить, запушить)
  git push origin main
  git tag -a v0.9.0 -m "Релиз v0.9.0 — описание изменений"
  git push origin v0.9.0   # ← запускает workflow и публикацию Release
  ```
  Первый релиз — `v0.9.0` (опубликован, оба архива на месте).

## 41. Модель слияния бэкапов (WebDAV + импорт)

Единый движок слияния (`src/webdav-sync.js`) используется и для облачной синхронизации, и для ручного импорта в режиме «Объединить». Слияние на уровне канала — `mergeChannelBackupData`, карты каналов — `mergeChannelsMaps`.

* **Статус и заметки тайтла** — LWW по `user_data[title].updatedAt` (бампится при любом изменении тайтла).
* **Прочитанные главы (`readPosts`) — tombstone-логика (`mergeReadState`).** У каждого поста есть время отметки прочтения (`readMarks[id]`) и время снятия (`unreadMarks[id]`); итог по посту — событие, произошедшее позже. Это синхронизирует **снятие** отметки между устройствами (чистый union этого не умел). Запись отметок делает `setPostReadState` (`src/state.js`), вызывается из обоих чекбокс-обработчиков сайдбара и из merge при переименовании тайтла (`grouping.js`). **Backward-compat:** legacy-данные (только массив `readPosts`, без меток) считаются прочитанными со времени `updatedAt` (базис 1) — побеждают при отсутствии конкурирующего tombstone, но проигрывают реальному снятию (`ts = Date.now()`). Явные `readMarks` хранятся только для постов, тронутых после апгрейда (без раздувания хранилища).
* **Настройки (`settings`) — LWW по `settings.updatedAt`.** Таймстамп бампится в `saveStateToStorage` только при реальном изменении пользовательских настроек (отпечаток `settingsFingerprint`, исключающий эфемерные `sidebarOpen`/сам `updatedAt`). Раньше выбор шёл по channel-level `exportDate`, который регенерируется при каждом экспорте → локальные настройки всегда «проигрывали» и терялись.
* **Списки «Новое» (`newTitles`/`newChapters`) — LWW целиком по `newListsUpdatedAt`** (отпечаток `newListsFingerprint` в `saveStateToStorage`). Пропагандирует и появление новинок, и их очистку. Per-item tombstone избыточен — новинки эфемерны и пересчитываются при каждой синхронизации постов.
* **Посты (`mergePosts`)** — по `id`, базовый берётся по `publishTime`. `isLiked` сливается через `||` (OR). **Сознательный компромисс:** снятый на одном устройстве лайк может временно вернуться, но `isLiked` авторитетно перетирается при следующей синхронизации постов из API Boosty (один аккаунт → обе стороны сходятся к серверу). Полноценный tombstone для лайков не нужен.
* **`playerTimestamps`** — `Math.max` по ключу: пересмотр с начала не «откатит» позицию (хранится максимум). By design.
* **`collapsedGroups`** — поверхностный merge (`{...local, ...remote}`), без таймстампа: чисто UI-состояние.
* **`blogDescriptionLinks`** — по `url` (Map), при дубле побеждает последний.

### 41.1 Условная запись по ETag (защита от lost update)
`provider.download()` возвращает `{ buffer, etag }`; `provider.upload(buffer, { etag })` при наличии etag шлёт `If-Match` — сервер отклонит запись (412), если файл в облаке изменился с момента чтения. `performWebDavSync` крутит цикл `download → merge → upload` и на 412 (`err.preconditionFailed`) перечитывает и сливает заново (до 3 попыток), защищая от потери чужих правок при одновременной синхронизации с двух устройств. В локальное хранилище/`state` изменения пишутся **только после успешной выгрузки** (`applyMergedChannelsToStorage` после цикла). ETag пробрасывается из `background.js` (`response.headers.get('etag')`). Если сервер ETag не отдаёт — запись безусловная (как раньше), регрессии нет.

### 41.2 Ручной импорт (`importBackupFile(file, mode)`)
Файловый ввод сайдбара показывает диалог выбора режима (`showImportChoiceDialog`):
* **`merge`** (по умолчанию, рекомендуется) — каналы из архива сливаются с текущими тем же движком (`mergeChannelBackupData`), ничего не теряется.
* **`replace`** — каналы из архива полностью перезаписывают текущие.

Оба режима нормализуют каждый `progress.json` через `normalizeImportedChannel` (валидация: `user_data` — объект, `posts`/массивы — массивы; битый JSON канала пропускается) и применяются через `applyMergedChannelToState` (каноничная форма хранения + накат в `state` для текущего канала + сброс снимков `settings`/новинок + обновление бейджа).

### 41.3 Передача тела WebDAV через прокси (base64)
WebDAV-запросы из content-скрипта проксируются через `background.js` (обход CORS/CSP — WebDAV-серверы не отдают CORS-заголовки, прямой кросс-origin `fetch` из content невозможен). `chrome.runtime.sendMessage` сериализует сообщение как **JSON**, поэтому `ArrayBuffer`/`Uint8Array` пересылку не переживают. Тело ZIP кодируется в **base64-строку** (`arrayBufferToBase64`/`base64ToArrayBuffer` в `src/utils.js`; в `background.js` те же хелперы продублированы инлайном, т.к. SW — не модуль). Кодирование чанками по `0x8000` байт — иначе `String.fromCharCode(...)` переполняет стек на больших архивах. Base64 (~1.33× данных) в разы экономнее по памяти и скорости сериализации, чем прежний массив байт-чисел (~5–10×), и снимает риск отказа `sendMessage` на крупных бэкапах. Прямой (не проксируемый) путь провайдера работает с `arrayBuffer` напрямую и кодирования не требует.

## 42. Фильтрация и условное логирование сетевых ошибок (Засорение Chrome Error Dashboard)

При возникновении штатных сетевых ошибок (например, `TypeError: Failed to fetch` при недоступности API Boosty или сбое интернет-соединения) передача объекта ошибки `e`/`err` в `console.error` или `console.warn` приводила к тому, что браузер Chrome регистрировал эти события как критические сбои расширения и отображал их в панели `chrome://extensions` (с кнопкой "Ошибки" и детальным стек-трейсом).

Для решения этой проблемы в релизных сборках (`DEV === false`) логирование сетевых сбоев и ошибок парсинга заменено на обычные вызовы `console.log` с выводом только текстового сообщения (`e.message || e`). В отладочных сборках (`DEV === true`) подробные `console.error` и `console.warn` с объектом ошибки сохранены для удобства отладки.

Изменения внесены в:
- `src/sync.js`: `syncBlogDescription`, `performIncrementalSync`, `performFullSync`, `backgroundSync`, `sendBoostyReaction`, `removeBoostyReaction`, `getWebDavOrigin`, `performWebDavSync`.
- `src/ui/sidebar.js`: обработка исключений при получении версии манифеста расширения.

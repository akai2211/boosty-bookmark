# Технические детали реализации функций: Boosty Bookmark

В данном документе собраны подробности технической реализации дополнительных возможностей расширения, которые были интегрированы на этапе развития интерфейса и настроек. Это позволяет сохранить контекст разработки и упростить дальнейшую поддержку кода.

---

## 1. Вкладка «Настройки»
Вкладка интегрирована в боковую панель как отдельный экран, сменяющий основной список тайтлов при переключении состояния `state.ui.activeTab === 'settings'`.

*   **Логика переключения:** Управление отображением происходит в функции `render()` в `content.js`. При активации вкладки вызывается функция `renderSettingsContent()`, которая заменяет содержимое контейнера `#lf-body-content`.
*   **Сохранение настроек:** Все изменения параметров (`syncLikes`, `autoMarkOpen`, `zoom`, `tabOrder`) немедленно сохраняются в локальное хранилище Chrome (`chrome.storage.local`) через функцию `saveStateToStorage()` и вступают в силу без перезагрузки страницы.

---

## 2. Экспорт и импорт прогресса (ZIP-архив)
Реализован ручной механизм резервного копирования и переноса прогресса пользователя (статусы, заметки в блокноте, просмотренные посты, настройки).

*   **Экспорт (`exportUserData`):** 
    Запрашивает все данные из локального хранилища `chrome.storage.local` (для поддержки бэкапа сразу по всем каналам автора). Используя библиотеку `JSZip`, формирует структуру папок для каждого канала (название папки совпадает с `slug` канала, например `lightfoxmanga`). Внутри каждой папки создается файл `progress.json`, который содержит версию структуры данных, дату экспорта, кэш постов (`posts`), прогресс пользователя (`user_data`), настройки расширения (`settings`), свернутые группы и ссылки профиля. Генерирует и инициирует скачивание ZIP-архива `boosty_bookmark_backup_YYYY-MM-DD.zip`.
*   **Импорт (`importUserData`):** 
    Принимает ZIP-архив, считывает его как `ArrayBuffer` и распаковывает средствами библиотеки `JSZip`. Проходит по всем файлам архива вида `имя_канала/progress.json`, считывает их и производит базовую валидацию. Все импортированные данные каналов сохраняются в локальное хранилище расширения (`lf_state_<slug>`). Если в импортируемом бэкапе присутствуют данные текущего активного канала, они немедленно загружаются в оперативную память расширения (`state`), после чего интерфейс обновляется.

## 3. Динамический порядок вкладок
Порядок вкладок больше не зашит жестко в разметке и рендерится динамически на основе массива настроек.

*   **Хранение:** Порядок хранится в `state.settings.tabOrder` в виде массива ключей вкладок.
*   **Отрисовка:** Функция `render()` обходит массив `state.settings.tabOrder` и генерирует HTML-код кнопок вкладок с соответствующими именами из словаря `TAB_NAMES`.
*   **Управление кнопками:** Рядом с каждым элементом в настройках выводятся кнопки ▲ и ▼. Функция `moveTab(index, direction)` меняет элементы массива местами, проверяя границы (блокируя кнопки `disabled` для крайних элементов), сохраняет состояние и вызывает полный ререндер интерфейса.

## 4. Нативный Drag and Drop для сортировки вкладок
Для более удобной сортировки вкладок в настройках реализован механизм Drag and Drop с помощью стандартного HTML5 Drag and Drop API.

*   **Безопасность перетаскивания (Draggable):** 
    Чтобы избежать случайного перетаскивания кнопок или текста, свойство `draggable="true"` присваивается элементу списка `.lf-tab-order-item` динамически только при событии `mousedown` на ручке перетаскивания `.lf-drag-handle` (иконка с 12 точками). При событиях `mouseup` и `dragend` перетаскиваемость сбрасывается в `false`.
*   **События Drag & Drop:** 
    *   `dragstart` — запоминает индекс перетаскиваемой вкладки.
    *   `dragenter` / `dragleave` — добавляет/удаляет класс `.lf-drag-over` для подсветки целевого элемента.
    *   `dragover` — отменяет стандартное поведение браузера для разрешения сброса (`drop`).
    *   `drop` — вызывает функцию `dragAndDropReorder(fromIndex, toIndex)`.
*   **Перестроение массива:** 
    Функция `dragAndDropReorder` вырезает элемент из исходного индекса через `splice` и вставляет его на новый целевой индекс, после чего сохраняет настройки и обновляет UI.
*   **CSS оптимизация:** 
    Для дочерних элементов внутри перетаскиваемой строки в CSS прописано `pointer-events: none` (кроме ручки и кнопок), чтобы при перетаскивании браузер не генерировал ложные события ухода мыши (`dragleave`) с родительского контейнера.

## 5. Сворачиваемые секции в настройках
Для экономии пространства на вкладке настроек блоки «Синхронизация и бэкап» и «Порядок вкладок» сделаны сворачиваемыми (collapsible).

*   **Состояние видимости:** За развернутое состояние секций отвечают переменные `state.ui.syncBackupExpanded` и `state.ui.tabOrderExpanded` соответственно (хранятся во временном состоянии интерфейса, сбрасываясь при закрытии страницы).
*   **Рендеринг и CSS:** 
    *   При значении `true` к контейнерам соответствующих секций добавляется класс `.lf-expanded`.
    *   По клику на шапки секций (`#lf-toggle-sync-backup` и `#lf-toggle-tab-order`) состояние инвертируется и запускается `render()`.
    *   В `styles.css` тело секции `.lf-settings-section-body` по умолчанию скрыто (`display: none`) и раскрывается в `display: block` только при наличии класса `.lf-expanded` у родительской секции.
    *   Иконка стрелочки `.lf-collapse-arrow` плавно поворачивается на 180 градусов благодаря `transition: transform 0.2s`.

## 6. Масштабирование панели (Zoom)
Позволяет пользователям кастомизировать масштаб выдвижной панели под разрешение экрана.

*   **Настройка масштаба:** В секцию интерфейса добавлен селектор выбора масштаба (от 80% до 150%). Значение записывается в `state.settings.zoom` (по умолчанию в интерфейсе установлено значение 100%, что физически соответствует коэффициенту 1.25).
*   **Применение:** При создании панели в функции `createSidebar()`, а также при изменении селектора в реальном времени, к DOM-элементу панели `#lf-sidebar` применяется свойство `style.zoom = state.settings.zoom / 100`. Это обеспечивает масштабирование всех дочерних элементов и шрифтов силами самого браузера.

## 7. Поведение свертывания подкатегорий (групп тайтлов)
Обеспечивает удобную навигацию по большому числу тайтлов на вкладках «Все» и «Новые» за счет их свертывания по умолчанию.

*   **Состояние свернутости по умолчанию:** Переменная `isCollapsed` в функции `renderGroup` вычисляется как `state.collapsedGroups[groupName] !== false`. Это означает, что если группа ранее не была явно развернута пользователем (её значение в объекте не равно `false`), она считается свернутой.
*   **Инверсия логики переключения:** При клике на заголовок группы, если она сворачивается, устанавливается `state.collapsedGroups[groupName] = true`, а если разворачивается — `state.collapsedGroups[groupName] = false`.
*   **Сброс при полной синхронизации:** Во время ручного запуска полной синхронизации базы постов (`performFullSync`) объект `state.collapsedGroups` принудительно очищается (`{}`), благодаря чему после обновления все категории автоматически возвращаются в исходное свернутое состояние.

## 8. Защита от ошибки «Extension context invalidated»
При обновлении или перезагрузке расширения в Chrome старый content-скрипт, уже внедрённый в страницу, продолжает выполняться. Однако его контекст (`chrome.runtime`, `chrome.storage`) становится недействительным, и любой вызов Chrome API выбрасывает ошибку `Uncaught Error: Extension context invalidated`. Это приводило к невозможности обновить расширение без его полного удаления.

*   **Функция `isExtensionContextValid()`:** Проверяет доступность контекста через `chrome.runtime && chrome.runtime.id`. Если `chrome.runtime.id` отсутствует или выбрасывает исключение, контекст считается мёртвым. Вызывается перед каждым обращением к Chrome API.
*   **Функция `cleanup()` и отписка от событий:** Останавливает периодический интервал проверки URL, удаляет все DOM-элементы расширения и **отписывается от всех глобальных слушателей событий** (`popstate`, `hashchange`, `click`, `beforeunload`), ссылки на которые хранятся в объекте `eventHandlers`. Это предотвращает срабатывание "мертвых" функций при навигации пользователя. Также восстанавливаются оригинальные методы `history.pushState` и `history.replaceState`.
*   **Безопасные обработчики:** Во все обработчики событий добавлена проверка `isExtensionContextValid()`. При попытке браузера вызвать обработчик из инвалидированного контекста, происходит вызов `cleanup()` и мгновенный возврат. Вызовы `window.dispatchEvent` дополнительно обернуты в `try/catch`.
*   **Обёртки Chrome API:** Все вызовы `chrome.storage.local.get/set` (в `loadStateFromStorage`, `saveStateToStorage`, обработчик `beforeunload`) обёрнуты в проверку контекста + `try/catch`, чтобы предотвратить необработанные исключения.
*   **Проверка `chrome.runtime.lastError`:** В колбеках `chrome.storage` дополнительно проверяется `chrome.runtime.lastError`, чтобы предотвратить ошибки, возникающие если контекст инвалидируется между вызовом и колбеком.

## 9. Синхронизация статуса лайков (isLiked)
Для того чтобы API возвращало актуальный статус пролайканных постов (поле `isLiked: true`), запросы к API (`https://api.boosty.to/...`) должны совершаться с использованием текущей сессии пользователя. 

*   **Авторизация через токен:** Одних только cookies для API Boosty (cross-origin запросы с `boosty.to` на `api.boosty.to`) оказалось недостаточно. API требует передачи токена доступа.
*   **Решение (getBoostyAuthToken):** В контент-скрипт добавлена функция `getBoostyAuthToken`, которая считывает токен авторизации пользователя (`accessToken`) из `window.localStorage` (ключ `auth`), где Boosty хранит активную сессию.
*   **Использование:** При каждом вызове `fetch` (как в `performFullSync`, так и в `backgroundSync`) этот токен подставляется в HTTP-заголовки запроса в виде `Authorization: Bearer <token>`. Это заставляет API Boosty распознавать пользователя и корректно отдавать статус `isLiked` для каждой главы произведений.

## 10. Стилизация чекбоксов просмотренных глав
Для улучшения видимости в темной и светлой темах, цвет чекбоксов изменен с системного серого на оранжевый (в стиле Boosty).

*   **Проблема с атрибутом `disabled`:** Ранее чекбоксы пролайканных на Boosty глав имели атрибут `disabled`, что заставляло браузер (особенно Chrome) принудительно окрашивать их в блеклые, нечитаемые цвета, игнорируя любые CSS-свойства вроде `accent-color`.
*   **Решение:** Атрибут `disabled` был удален. Вместо него пролайканные главы получают класс `.lf-liked-checkbox`.
*   **Окрашивание:** Для всех чекбоксов глав (`.lf-chapter-checkbox`) установлено свойство `accent-color: var(--lf-primary);`, что делает активные галочки оранжевыми во всех темах браузера.
*   **Снятие лайков:** Ранее чекбоксы пролайканных постов блокировались (через `pointer-events: none` и JS). Теперь эта блокировка снята: пользователь может свободно снимать галочки. При снятии галочки отправляется `DELETE` запрос на API Boosty (`removeBoostyReaction`), а кэш поста обновляется.

## 11. Сохранение состояния открытости панели и рендеринг при инициализации
Для устранения бага, когда после обновления страницы (F5) боковая панель могла отображаться пустой (только размытый фон backdrop), были внесены следующие изменения:

*   **Сохранение открытости (`sidebarOpen`):** 
    В настройки расширения (`state.settings.sidebarOpen`) добавлено логическое свойство, которое сохраняется в локальное хранилище Chrome при клике на кнопку-триггер, клике снаружи для закрытия или при автоматическом скрытии при уходе с целевой страницы.
*   **Восстановление состояния:** 
    При создании контейнера панели в `createSidebar()` проверяется сохраненное значение `state.settings.sidebarOpen`. Если оно равно `true`, к боковой панели сразу добавляется класс `.lf-open` для её плавного выдвижения.
*   **Принудительный рендеринг при старте:** 
    Ранее при наличии постов в кэше (`state.posts.length > 0`) контент-скрипт запускал фоновое обновление `backgroundSync()`, но не вызывал функцию `render()`. Из-за этого при перезагрузке страницы DOM-структура панели оставалась пустой, пока не приходил ответ от API Boosty с какими-либо обновлениями. Теперь при наличии постов функция `render()` вызывается незамедлительно при создании интерфейса, заполняя сайдбар сохраненным в памяти кэшем глав, и только затем запускается фоновый запрос обновлений.

## 12. Позиционирование панели и логика скрытия кнопки-триггера
Для того чтобы интерфейс расширения не перекрывал шапку (header) сайта Boosty, и для улучшения UX при открытии сайдбара, были внесены следующие изменения:

*   **Вертикальный сдвиг сайдбара с учетом масштабирования (zoom):**
    В CSS для контейнера `#lf-sidebar` заданы свойства `top: 56px;` (координаты fixed-позиционирования не масштабируются браузером при применении `zoom`, поэтому отступ сверху всегда составляет физические `56px`) и `height: calc((100vh - 56px) / var(--lf-zoom-val));` (деление на коэффициент масштабирования компенсирует растяжение высоты). Это гарантирует, что при любом масштабе сайдбар на экране располагается ровно под шапкой Boosty, не перекрывая элементы хедера, а его физическая высота остается равной доступной высоте вьюпорта, что сохраняет работоспособность внутреннего скроллбара.
*   **Скрытие плавающей кнопки-триггера:**
    В CSS добавлен декларативный селектор соседа `#lf-sidebar.lf-open ~ #lf-trigger-btn { display: none !important; }`. При открытии панели кнопка-триггер автоматически скрывается, разгружая интерфейс.
*   **Порядок создания элементов в DOM:**
    В функции `checkUrlAndToggleVisibility()` изменен порядок вызовов: теперь сначала вызывается `createSidebar()`, а затем `createTriggerButton()`. Это гарантирует, что кнопка-триггер идет после сайдбара в дереве DOM, что необходимо для корректной работы CSS-селектора общего соседа (`~`).
*   **Кнопка закрытия (сворачивания) в шапке панели:**
    В верстку шапки панели (справа) добавлена круглая кнопка сворачивания `#lf-close-btn` с SVG-иконкой стрелочки вправо. При нажатии на нее панель плавно уезжает вправо, а состояние `state.settings.sidebarOpen` сбрасывается в `false`. Автоматическое закрытие панели при клике снаружи (потеря фокуса) было полностью отключено по просьбе пользователя для повышения стабильности UX.

## 13. Автоматическая постановка лайка (реакции) при отметке просмотренного
Для обеспечения бесшовной интеграции с экосистемой Boosty, при ручной установке галочки чекбокса в расширении теперь автоматически отправляется запрос на постановку лайка к этому посту.

*   **Endpoint API:** Запрос отправляется методом `POST` на `https://api.boosty.to/v1/blog/lightfoxmanga/post/<postId>/reaction?from_page=blog`.
*   **Авторизация:** Используется тот же токен (`accessToken`), получаемый из `localStorage` браузера пользователя.
*   **Payload:** Отправляется в формате `application/x-www-form-urlencoded` с телом `reaction=heart`, что соответствует стандартному лайку (сердечку) на Boosty.
*   **Логика:** Функция `sendBoostyReaction(postId)` вызывается только в случае установки галочки (`e.target.checked === true`). Локальный кэш поста немедленно обновляется (`isLiked = true`), чтобы избежать рассинхронизации при последующем рендере. Снятие галочки лайк не удаляет (по запросу пользователя).

## 14. Мгновенная синхронизация лайков через перехват fetch
Для обеспечения реактивности расширения без необходимости ручного обновления базы реализован перехват `window.fetch`. Когда пользователь ставит лайк на странице Boosty, расширение мгновенно обновляет свой кэш и интерфейс.

*   **Функция `patchFetch()`:**
    Сохраняет ссылку на оригинальный `window.fetch` в `eventHandlers.originalFetch` и подменяет его обёрткой-прокси. Обёртка прозрачно вызывает оригинальный fetch и параллельно анализирует URL и метод запроса.
*   **Условие перехвата:**
    Отлавливаются только `POST`-запросы, URL которых содержит `/reaction`. Из URL извлекается `postId` с помощью регулярного выражения `/\/post\/([a-f0-9-]+)\/reaction/i`. Это покрывает как запросы самого расширения (`sendBoostyReaction`), так и нативные запросы SPA-приложения Boosty при клике на кнопку лайка на сайте.
*   **Обработка результата (`handleInterceptedReaction`):**
    Вызывается асинхронно через `.then()` только при успешном ответе сервера (`response.ok`). Находит пост в кэше `state.posts` по `postId` и выставляет `isLiked = true`. Обновляет хранилище через `saveStateToStorage()`.
*   **Точечное обновление UI:**
    Если пользователь находится в детальном виде тайтла — ищет конкретный чекбокс по `data-post-id` и обновляет его состояние (`checked = true`, добавляет класс `lf-liked-checkbox`), а также пересчитывает счётчик прогресса в заголовке. Если пользователь в режиме списка — вызывает `renderListContent()` для обновления цветовых индикаторов.
*   **Восстановление в `cleanup()`:**
    При инвалидации контекста расширения оригинальный `window.fetch` восстанавливается из `eventHandlers.originalFetch`, чтобы не нарушать работу сайта Boosty после обновления расширения.

## 15. Двусторонняя синхронизация лайков в реальном времени (История решения)
**Задача:** Обеспечить мгновенное отображение изменений:
1. При постановке лайка на самом сайте Boosty галочка в расширении должна появляться без обновления страницы.
2. При постановке/снятии галочки в расширении, лайк на самом сайте Boosty должен появляться/исчезать без обновления страницы.

**Попытка 1: DOM-синхронизация (Программный клик - ОТКЛОНЕНО)**
Была попытка реализовать двустороннюю связь через поиск кнопки лайка в DOM и вызов `likeBtn.click()` из расширения.
*Что пошло не так:*
- `querySelector` цеплял ссылки из нашего же сгенерированного расширением сайдбара.
- Основная проблема: кнопка `[data-test-id="COMMON_REACTIONS_REACTIONSPOST:ROOT"]` на Boosty не отправляет запрос на лайк напрямую. Её нажатие лишь открывает React-меню (tooltip) выбора реакций. Из-за этого программный клик эмулировал открытие меню, но реального API-запроса на сервер Boosty не происходило.

**Финальное решение: Внедрение скрипта в Main World + Перехват XMLHttpRequest/fetch**
Так как Content Scripts расширения работают в **isolated world**, они не могут перехватывать запросы, отправленные скриптами сайта в **main world**.
*   Был создан отдельный скрипт `page_script.js`, который прописан в `manifest.json` со свойством `"world": "MAIN"`.
*   Этот скрипт переопределяет нативные `window.fetch` и методы `XMLHttpRequest.prototype.open`/`send` на самой странице Boosty.
*   *Особенность Boosty:* Выяснилось, что для лайков сайт использует устаревший `XMLHttpRequest`, а не `fetch`.
*   При перехвате успешного POST/DELETE запроса к `/reaction`, `page_script.js` отправляет сообщение через `window.postMessage({ type: 'LF_REACTION_INTERCEPTED' ... })`.
*   Основной скрипт расширения `content.js` слушает эти сообщения (`window.addEventListener('message')`) и мгновенно обновляет локальный кэш и галочки в UI.
*   При клике на галочку в самом расширении мы больше не пытаемся "кликать" по сайту, а просто отправляем прямой `POST` или `DELETE` API-запрос на сервер с авторизационным токеном из LocalStorage.

## 16. Автоматический скролл к ленте постов
Для улучшения UX при переходе к конкретному тайтлу из карточки расширения реализован автоматический скролл мимо длинного блока «Об авторе» прямо к началу ленты постов (кнопкам «Лента» / «Медиа»).

*   **Триггер автоскролла (`checkAndScrollToFeed`):**
    Проверяет, содержит ли текущий URL параметры фильтрации Boosty (`postsTagsIds` или `tag`).
*   **Сравнение URL:**
    Чтобы предотвратить повторные нежелательные скроллы при прокрутке страницы пользователем, в замыкании сохраняется `lastScrolledUrl`. Скролл срабатывает только при переходе на новый уникальный URL.
*   **Ожидание элемента в DOM:**
    Поскольку Boosty работает как SPA, вкладки ленты могут отрисовываться с задержкой. Функция запускает интервал (до 3 секунд), опрашивающий DOM на наличие элемента `[class*="FeedTabs-scss--module_root_"]`.
*   **Смещение скролла:**
    Чтобы зафиксированная шапка Boosty (высотой 56px) не перекрывала кнопки «Лента / Медиа», прокрутка выполняется с вертикальным смещением в `-80px` от верхней границы элемента:
    `const y = rect.top + window.pageYOffset - 80; window.scrollTo({ top: y, behavior: 'smooth' });`
*   **Сброс состояния:**
    При вызове функции `cleanup()` переменная `lastScrolledUrl` сбрасывается в `null`.

## 18. Сохранение состояния интерфейса во вкладке через sessionStorage
Чтобы переходы в текущей вкладке (вызывающие полную перезагрузку страницы Boosty) не сбрасывали открытую карточку тайтла и текущую вкладку расширения, состояние UI временно кэшируется в `sessionStorage` браузера для этой вкладки.

*   **Сохранение состояния (`render`):**
    При каждом рендере интерфейса текущие значения `state.ui.activeTitle` и `state.ui.activeTab` записываются в `sessionStorage`:
    `sessionStorage.setItem('lf_active_title', state.ui.activeTitle || '');`
    `sessionStorage.setItem('lf_active_tab', state.ui.activeTab || 'favorite');`
*   **Восстановление состояния (`init`):**
    При инициализации расширения (после перезагрузки страницы или первоначальной загрузки) расширение считывает эти значения и восстанавливает их в `state.ui`. Если в `sessionStorage` был сохранен активный тайтл, расширение сразу рендерит детальный вид этого тайтла.
*   **Сброс при клике «Назад»:**
    Когда пользователь нажимает «Назад к списку», `activeTitle` сбрасывается в `null`, `render()` перезаписывает сохраненное значение пустой строкой, что гарантирует правильное поведение при последующих переходах.
*   **Очистка в `cleanup`:**
    При полной выгрузке расширения/инвалидации контекста ключи удаляются через `sessionStorage.removeItem`.

## 19. Бесшовная SPA-навигация при переходе по ссылке тега
Чтобы исключить полную перезагрузку страницы при клике на тег тайтла (которая вызывает мерцание, уничтожение DOM расширения и повторную его загрузку), в детальном виде реализован перехват события клика.

*   **Перехват клика (`renderDetailContent`):**
    На элемент `.lf-detail-title-link` вешается обработчик события `click`.
*   **Исключения (открытие в новой вкладке):**
    Если отключена настройка `openTitlesInCurrentTab` или пользователь зажал клавиши-модификаторы (Ctrl/Cmd) или нажал на колесико мыши (средний клик), обработчик позволяет выполнить стандартное поведение браузера (открытие в новой вкладке).
*   **Плавный SPA-переход:**
    В остальных случаях срабатывает `e.preventDefault()`, и переход осуществляется средствами React Router Boosty:
    1. Через `history.pushState({}, '', relativeUrl)` меняется адресная строка.
    2. Вызовом `window.dispatchEvent(new PopStateEvent('popstate', { state: {} }))` React Router сайта уведомляется о переходе, что заставляет его перерисовать ленту постов.
    3. Это вызывает нативное событие `lf_locationchange` расширения, и плавно скроллит страницу к началу ленты без перезагрузки всей страницы и закрытия сайдбара.

## 20. Сброс всех данных расширения
Позволяет полностью очистить локальное хранилище расширения и вернуть его к первоначальному состоянию.

*   **Интерфейс:**
    В самый низ вкладки настроек (под блоком «Инфо о расширении») добавлена скромная, невзрачная кнопка `#lf-delete-data-btn` (стилизована как второстепенная кнопка с блекло-красной границей и текстом, переходящими в насыщенный цвет при наведении).
*   **Подтверждение (Двухэтапное):**
    Во избежание блокировок браузера на вызовы нативных диалоговых окон (`confirm`) реализовано встроенное двухэтапное подтверждение:
    1. При первом клике кнопка меняет статус (`data-confirming="true"`), окрашивается в ярко-красный цвет, а её текст меняется на *«Вы точно уверены? Нажмите для удаления»*.
    2. Рядом динамически создается кнопка *«Отмена»* (при клике на неё кнопка возвращается в исходное состояние).
    3. Запускается таймер на 5 секунд: если повторный клик не совершен, кнопка автоматически сбрасывается в исходное состояние.
    4. При повторном клике по кнопке в активном состоянии подтверждения выполняется очистка данных.
*   **Логика очистки:**
    1. С помощью `chrome.storage.local.clear()` удаляются все данные из постоянного хранилища.
    2. Локальный объект состояния `state` сбрасывается к значениям по умолчанию.
    3. Очищаются временные данные в `sessionStorage` (`lf_active_title` и `lf_active_tab`).
    4. Масштаб боковой панели возвращается к дефолтному (100% / физический 1.25).
    5. Вызывается функция `render()`, которая перенаправляет пользователя на стартовый экран для выполнения первой синхронизации.

## 21. Переход и скролл к главе в ленте тайтла (Infinite Scroll)
Для повышения удобства чтения расширение позволяет переходить непосредственно к посту с выбранной главой в ленте тегов вместо открытия поста на отдельной странице.

При переключении чекбокса в состояние «включено» под строкой настройки динамически появляется блок с предупреждением о бета-статусе функции и крестиком закрытия. При отключении чекбокса предупреждение скрывается. В выключенном состоянии главы по умолчанию открываются как отдельные страницы постов (в текущей или новой вкладке, согласно настройке `openTitlesInCurrentTab`).

*   **SPA-переход для глав:**
    При переходе в текущей вкладке (`openTitlesInCurrentTab === true`) клик по ссылке главы всегда перехватывается: URL меняется через `history.pushState` (соответственно с хэшем или на прямую страницу поста), а событие `popstate` триггерит встроенный роутер Boosty для бесшовного SPA-перехода без перезагрузки страницы.

    *   **Скролл к элементу:** При нахождении ссылки вычисляется родительский контейнер поста `.closest('[class*="Post-scss--module_root"]')` и к нему совершается плавный скролл с отступом `-80px` под зафиксированную шапку Boosty.

## 22. Сортировка тайтлов на вкладках библиотеки
Для быстрого нахождения и упорядочивания произведений в боковой панели реализована интерактивная сортировка списка тайтлов.

*   **Интерфейс:**
    В строку поиска встроен flex-контейнер `.lf-search-row`. Справа от поля ввода размещена кнопка сортировки `#lf-sort-btn` с иконкой. При клике на неё раскрывается выпадающий список `#lf-sort-dropdown` с 8 режимами. Выбранный режим выделяется подсветкой `.lf-active`.
*   **Режимы сортировки (`state.settings.titleSort`):**
    *   `name_asc` / `name_desc` — По названию: А → Я / Я → А через сравнение `localeCompare`.
    *   `new_desc` / `new_asc` — По новизне (сначала новые / старые) на основе `publishTime` самого последнего поста в тайтле.
    *   `chapters_desc` / `chapters_asc` — По общему количеству глав (по убыванию / возрастанию).
    *   `progress_desc` / `progress_asc` — По прогрессу просмотра (сначала произведения с большим / меньшим количеством просмотренных глав, с дополнительной сортировкой по проценту просмотренного и алфавиту при совпадении).
*   **Сохранение и оптимизация:**
    Выбранный режим сортировки сохраняется в `state.settings.titleSort` и синхронизируется в `chrome.storage.local`. При изменении сортировки перерисовывается только список тайтлов (`renderListContent()`), что исключает потерю фокуса в поле поиска и обеспечивает моментальный отклик интерфейса.

## 24. Архитектура и анализ синхронизации (WebDAV / Google Диск)

В данном разделе приведён подробный анализ механизмов синхронизации в расширении и предложена гибкая **мультипровайдерная архитектура** для реализации синхронизации в **Boosty Bookmark**. 

Архитектура позволяет объединить преимущества нативной интеграции с Google Диском в Chrome и универсального стандарта WebDAV для кроссплатформенности (включая будущую поддержку Firefox).

### 24.1 Анализ ограничений chrome.storage.sync
Использование `chrome.storage.sync` накладывает жесткие лимиты со стороны Chrome API, что делает его непригодным для хранения больших объемов данных расширения:
- **QUOTA_BYTES** (Общий объем): **102 400 байт (100 КБ)**. Наша база постов (`state.posts` для 1200+ постов) весит ~300 КБ. Синхронизировать её целиком в `sync` физически невозможно.
- **QUOTA_BYTES_PER_ITEM** (Размер ключа): **8 192 байта (8 КБ)**. Прогресс пользователя (`state.user_data`) хранится в виде одного объекта. При большом количестве тайтлов размер ключа превысит 8 КБ.
- **MAX_ITEMS** (Количество ключей): **512 ключей**.
- **Проблема кроссплатформенности:** `chrome.storage.sync` привязан к экосистеме конкретного браузера и не позволяет переносить данные между разными браузерами (например, Chrome и Firefox).

### 24.2 Мультипровайдерная архитектура
Для обеспечения свободы выбора пользователя и кроссплатформенности синхронизация проектируется по паттерну **«Стратегия» (Strategy)**. Вся логика работы с конкретным облаком скрывается за единым абстрактным интерфейсом `SyncProvider`.

*   **Google Drive (Нативный OAuth2):** Использует API `chrome.identity.getAuthToken` для бесшовной интеграции в один клик. Данные хранятся в скрытой папке приложения `appDataFolder` на диске пользователя.
*   **WebDAV (Яндекс.Диск, Nextcloud и др.):** Использует стандартные HTTP-запросы с базовой авторизацией (Basic Auth) по логину и паролю приложения.

### 24.3 Формат файла синхронизации (`boosty_bookmark_backup.json`)
Мы **не синхронизируем** локальный кэш постов (`state.posts`), так как он весит слишком много и его можно загрузить по API Boosty. Синхронизации подлежат только:
- Настройки расширения (`state.settings`).
- Прогресс прочитанных глав и отслеживаемых тайтлов (`state.user_data`).
- Сохраненные таймстампы медиаплееров (`state.playerTimestamps`).

```json
{
  "version": 1,
  "updatedAt": 1781423856000,
  "settings": {},
  "user_data": {},
  "playerTimestamps": {}
}
```

### 24.4 Слияние данных (Data Merging Strategy)
Поскольку бэкап может обновляться на разных устройствах асинхронно, используется стратегия слияния на основе временных меток (timestamps):
1. **Настройки:** Приоритет отдается настройкам с более поздним `updatedAt`.
2. **Прогресс постов (`user_data`):** Списки прочитанных глав объединяются (операция `Union`). Если глава была помечена как прочитанная на одном устройстве, она станет прочитанной и на другом.
3. **Таймстампы плееров:** Для каждого медиа-файла сохраняется таймстамп с более поздней датой последнего изменения.

## 25. Принудительное изменение качества видео (Force Video Quality) в плеере VK

В данном разделе описан механизм автоматической установки качества видео в веб-компоненте `<vk-video-player>`. Вся внутренняя структура плеера изолирована в **Shadow DOM** этого элемента.

### 25.1 Структура DOM плеера
1. Веб-компонент: `<vk-video-player>`
2. Контейнер со скрытым DOM: `div.shadow-root-container` (содержит `#shadow-root`)
3. Внутри Shadow Root:
   - Главная обертка плеера: `div.player-wrapper`
   - Контейнер интерактива: `div.container` (внутри `div.player-wrapper`)
   - Элемент видео: `<video>`
   - Меню настроек качества (появляется в разметке управления):
     - Элементы списка: `li.item-quality` (содержат `data-value` со значением разрешения, например `"1080p"`).

### 25.2 Алгоритм работы функции
Процесс принудительной установки качества разделен на три шага:
1. **Обнаружение плеера и доступ к Shadow DOM:** Скрипт находит контейнер `.shadow-root-container` и обращается к свойству `.shadowRoot`.
2. **Привязка к первому клику:** Поскольку элементы управления генерируются динамически при запуске, скрипт вешает однократный обработчик клика (`{ once: true }`) на элемент `div.player-wrapper div.container`. Это позволяет применить качество в момент старта видео.
3. **Программный клик по пункту качества:**
   - Производится поиск всех элементов `li.item-quality`.
   - Если найдено совпадение с выбранным пользователем качеством (например, `1080p`), выполняется `qualityEl.click()`.
   - Если точного совпадения нет, выбирается первый доступный вариант (обычно максимальный).

*(Примечание: в текущей версии расширения данная функция временно отключена из-за перехода плеера на двухуровневое меню и ленивую прорисовку элементов управления).*

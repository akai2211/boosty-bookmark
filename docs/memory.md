# Память агента: Boosty Bookmark

Данный файл содержит важные правила, технические нюансы и предпочтения пользователя, обязательные для соблюдения во всех последующих сессиях разработки.

## 1. Языковые настройки и стиль общения

*   **Стиль принятия решений:** Никогда не спешить. Всегда запрашивать явное согласие пользователя (Акая) перед важными изменениями.

## 2. Правила работы с Git и тестированием (КРИТИЧЕСКИ ВАЖНО)
*   **Запрет на автоматический push:** **НИКОГДА** не выполнять команду `git push` или любые другие команды, отправляющие изменения в удаленный репозиторий, автоматически. Это действие выполняется **только** после прямого указания Акая (*«сделай пуш»*, *«запушь»*).
*   **Запрет на автоматический коммит:** **НИКОГДА** не делать локальные коммиты (`git commit`) без прямого указания Акая (*«закоммить»*, *«сделай коммит»*).
*   **Правило запуска тестов:** Запуск тестов (`npm test`, `playwright`, `vitest` и др.) разрешен **только перед коммитом** (после получения команды закоммитить) для валидации изменений перед коммитом. В остальных случаях запускать тесты без прямой необходимости запрещено.
*   **Информационные действия разрешены:** Выполнение вспомогательных команд `git add` (для индексации), `git status`, `git diff`, `git log` разрешено без предварительного согласования.
*   **Верификация актуальности памяти:** Перед началом работы над любой задачей (в новой сессии или чате) ИИ-ассистент **обязан** сверить содержимое файлов памяти (`docs/memory.md` и глобального KI) с реальным кодом проекта. При обнаружении расхождений (например, функция числится в планах, но уже написана) необходимо сначала предложить пользователю исправление расхождений и актуализировать память, и только потом приступать к реализации новых функций.
*   **Правило обновления памяти и коммита:** При получении команды от Акая *«обнови память и закоммит»* (или аналогичной), ИИ-ассистент обязан автоматически актуализировать все файлы документации в папке `docs/` (внести выполненные задачи в `tasks.md`, добавить описание изменений в `walkthrough.md`/`technical_notes.md` и добавить запись о новом коммите в раздел `5. Выполненные коммиты` в `memory.md`), запустить тесты для проверки работоспособности и только после этого сделать локальный коммит (без автоматического push).

## 4. Архитектура и технические особенности

### Интерфейс (UI/UX)
*   **Кнопка-триггер (Закладки):** Спозиционирована по вертикали ровно по центру экрана (`top: 50%; transform: translateY(-50%)`). Внутренняя иконка представляет собой белую закладку с вырезанной молнией (размер в CSS `28x34px`, `viewBox="550 450 850 1020"`). Для предотвращения перекрашивания иконки в темный цвет в светлой теме Boosty, путь закладки в SVG жестко залит белым цветом (`fill="#ffffff"` вместо `fill="currentColor"`).
*   **Логотип сайдбара (Инверсия):** Из логотипа сайдбара (в шапке и на пустом экране) полностью убрана оранжевая квадратная плашка. Оставлена только белая закладка с оранжевой молнией. Молния закрашивается за счет внутренней оранжевой подложки (`rect` с координатами `x="700" y="500" width="450" height="630"`). При этом:
    * Высота подложки уменьшена с `650` до `630`, чтобы нижний полукруглый вырез закладки оставался прозрачным.
    * Для светлой темы сайдбара (`#lf-sidebar.lf-light`) настроена автоматическая инверсия через CSS: закладка закрашивается оранжевым градиентом, а молния — белым цветом, сохраняя высокую читаемость.
*   **Автоматическое скрытие сайдбара:** Добавлен глобальный обработчик кликов. Сайдбар автоматически закрывается при клике на колокольчик уведомлений (`button[class*="NotificationsButton-scss--module_button_"]`) или на аватар пользователя / меню в шапке Boosty (`div[class*="MiniProfile-scss--module_root_"][role="button"]`), чтобы не мешать открывающимся выпадающим меню сайта.
*   **Позиционирование:** Сайдбар сдвинут по высоте на 56px вниз (`top: 56px; height: calc((100vh - 56px) / var(--lf-zoom-val))`), прилегая к хедеру Boosty.
*   **Порядок вкладок:** По умолчанию: Избранное, Новое, Смотрю, Все, а затем выпадающая кнопка «Архив» (с завершенными/брошенными). Предусмотрена автоматическая миграция старых порядков на новый.
*   **Архив (дропдаун):** Вкладки «Завершено» и «Брошено» объединены в дропдаун. При выборе одной из них кнопка меняет название (например, «Завершено ▼» или «Брошено ▼») и подсвечивается как активная.
*   **Масштабирование:** Масштаб по умолчанию — 100%, который физически равен коэффициенту 1.25. Все опции масштабирования хранятся в БД в виде числовых коэффициентов (например, 80% -> 1.0, 100% -> 1.25, 150% -> 1.875) и напрямую передаются в CSS-переменную `--lf-zoom`. Реализована автоматическая однократная миграция старых строковых или процентных значений в коэффициенты с флагом `zoomMigrated`.
*   **Кнопка настроек:** Шестеренка работает как переключатель (toggle): повторный клик возвращает пользователя на ту вкладку, на которой он был ранее.
*   **Предотвращение сброса прокрутки:** При любых перерисовках сайдбара (сортировка вкладок, переключение настроек, изменение масштаба) текущее значение `scrollTop` контейнера `#lf-body-content` сохраняется и восстанавливается автоматически, предотвращая нежелательные прыжки скролла наверх.
*   **Раздел «О расширении»:** Доступен по кнопке в самом низу вкладки настроек. Экран полностью перекрывает настройки и выполнен в супер-компактном виде. Содержит имя автора (**Akai**), сведения о лицензии (**MIT**), конфиденциальности (**Локально или WebDAV**) и дисклеймер. Ссылка обратной связи перенаправлена на Boosty Акая (`https://boosty.to/akai2211`). Блок поддержки переработан:
    *   **Ряд 1**: Приоритетная кнопка Boosty (Подписка и Новости) на всю ширину со звездочкой, ведущая на `https://boosty.to/akai2211`.
    *   **Ряд 2 (на всю ширину)**: Кнопка USDT (TRC-20) с иконкой доллара, растянутая на 100% ширины строки.
    *   **Удаленные методы**: YooMoney, DonationAlerts и TON полностью исключены из интерфейса поддержки.
    *   **Интерактивный поп-ап USDT (TRC-20):** При клике на кнопку USDT открывается модальное окно `#lf-usdt-modal` с эффектом `backdrop-filter: blur(4px);` поверх сайдбара. Окно содержит QR-код (Base64), предупреждение о необходимости использования сети TRC-20, адрес кошелька с подписью **«Адрес USDT (TRC-20)»** / **«USDT (TRC-20) Address»** и кнопки копирования.
    *   **Механизмы копирования:** Адрес кошелька копируется по клику на основную кнопку копирования, на встроенную иконку копирования справа от адреса, а также по двойному клику на текстовое поле адреса или изображение QR-кода. Копирование вызывает глобальный тост «Адрес скопирован!» и подсвечивает кнопки зеленым цветом на 2 секунды. Модальное окно плавно скрывается и удаляется из DOM по клику на крестик или оверлей.



### Бизнес-логика и парсинг
*   **Авто-статус:** Тайтлу автоматически присваивается статус «Смотрю» (`watching`), если счетчик прочитанных глав больше 1, а текущий статус равен `none`. Для исключения конфликтов с WebDAV, при автоматическом создании статуса `updatedAt` устанавливается в `0`, а при автообновлении существующего статуса его `updatedAt` сохраняется.
*   **Автоперенос в завершенные:** Если тайтл полностью озвучен (`isFullyFinished === true`), все посты просмотрены, а статус тайтла равен `none` или `watching`, статус автоматически переключается на `completed` (без изменения `updatedAt`).
*   **Группа «Завершен том» во вкладке «Смотрю»:** Если у тайтла со статусом `watching` просмотрены все главы и при этом у него завершен том (`isVolumeFinished === true`), он автоматически убирается из основного списка «Смотрю» и опускается в свернутую по умолчанию группу «Завершен том» внизу вкладки.
*   **Категория подписки:** В детальном виде тайтла выводится уровень подписки (например, «Для шейхов») прямо под заголовком.
*   **Спецкатегории завершенности:** Тайтлы дублируются в спецкатегории «Полностью озвучено» или «Завершен том» на вкладке «Все». Для надежного определения:
    1. Парсится текстовое примечание (`note`), следующее за ссылкой в описании профиля (например, `[Полностью озвучен, конец 1 тома] 🔥`).
    2. Проверяется заголовок самого последнего поста тайтла (если вышли новые посты после отметки о завершении тома, тайтл считается продолжающимся). Ложные смайлики `🔥` в обычных главах игнорируются.
*   **Синхронизация названий (Красивые имена):** Расширение парсит ссылки (теги `postsTagsIds` и ссылки на посты) в поле описания профиля автора ("Об авторе") и связывает их с оригинальными названиями из текста ссылок. Пользовательский прогресс автоматически мигрирует на новые красивые имена. Если в процессе миграции обнаруживается, что в `state.user_data` уже существуют оба ключа (технический и красивый), выполняется их интеллектуальное объединение на основе таймстампов `updatedAt` для предотвращения дублирования и потери ручных записей.
*   **Подкатегория «Все» на вкладке «Все»:** Внутрь вкладки «Все» добавлена новая общая подкатегория `'Все'` на самой верхней позиции. Она представляет собой плоский отсортированный список всех тайтлов (без разбивки по уровням подписок), что упрощает глобальный поиск и сортировку.
*   **Категория «Бесплатные»:** Категория по умолчанию для бесплатного контента или постов без тега подписки переименована с `'Все'` на `'Бесплатные'` для предотвращения путаницы с общими списками и вкладкой.
*   **Снятие лимита на высоту списков (max-height):** Класс `.lf-group-list` в `styles.css` имел ограничение `max-height: 1000px;` для анимации раскрытия. Это приводило к обрезанию списков длиннее 27 элементов (остальные тайтлы скрывались свойством `overflow: hidden;`). Лимит увеличен до `99999px` для поддержки списков любого размера.
*   **Логика вкладки «Новые» и времени последнего визита (`lastVisit`):**
    * Категории упрощены до «Новые тайтлы» и «Новые главы», а группы на вкладке всегда принудительно раскрыты (`isCollapsed = false`).
    * **Сортировка:** Вкладка «Новые» не подчиняется глобальной настройке сортировки. Элементы внутри её групп («Новые тайтлы» и «Новые главы») всегда принудительно сортируются по дате публикации самого последнего поста/обновления по убыванию (`new_desc` / сначала новые). Это гарантирует, что свежие обновления всегда отображаются вверху списка и пользователь не пропустит их.
    * При обнаружении свежих обновлений после любой синхронизации (инкрементальной, полной или фоновой) расширение автоматически переключает активную вкладку на «Новые» (`state.ui.activeTab = 'new'`).
    * При включенной дев-эмуляции даты `state.lastVisit` инициализируется значением `cutoffTimeMs - 24 часа` при переходе из будущего/реального времени в прошлое (или при значении `0`), чтобы при первой синхронизации последние посты считались новыми. При изменении даты отсечки в DevTools вперёд значение `lastVisit` сохраняется (не сбрасывается в `0`), чтобы корректно эмулировать поступление новых постов. При сохранении настроек DevTools автоматически запускается инкрементальная синхронизация для моментального обновления UI.
    * Для защиты от автоматического исчезновения новинок (если пользователь обновил страницу, не открыв сайдбар) `state.lastVisit` обновляется в хранилище только при подтвержденном просмотре новинок пользователем. Это происходит при переключении с вкладки «Новые» на другую вкладку, сворачивании сайдбара кнопкой закрытия/триггера или закрытии страницы (`beforeunload`), но только при условии, что сайдбар был открыт и вкладка «Новые» была активна.


### Бэкап и многоканальность
*   **Бэкап в ZIP-архив (JSZip):** Для импорта и экспорта прогресса используется библиотека `JSZip` (подключена в `manifest.json` перед основным скриптом расширения).
*   **Многоканальный экспорт:** При нажатии кнопки бэкапа расширение обходит все сохраненные состояния каналов (`lf_state_<slug>`) из локального хранилища браузера и собирает их в ZIP-архив с иерархией `channel_name/progress.json`. Имя файла по умолчанию: `boosty_bookmark_backup_YYYY-MM-DD.zip`.
*   **Многоканальный импорт:** При импорте ZIP-архива данные по всем каналам извлекаются и записываются в хранилище Chrome, а текущий открытый канал мгновенно применяет изменения в память и обновляет UI.
*   **Сворачиваемый блок настроек:** Блок «Синхронизация и бэкап» в меню настроек сделан сворачиваемым (collapsible) и по умолчанию находится в свернутом состоянии для экономии интерфейсного места.

### Облачная синхронизация (WebDAV)
*   **Выбор провайдера:** В настройках реализован переключатель `webdavConfig.provider` со значениями `'yandex'` (по умолчанию) и `'webdav'`. При выборе Яндекс.Диска адрес сервера фиксируется на `https://webdav.yandex.ru` и поле ввода адреса скрывается. При выборе другого WebDAV отображаются поле адреса и инструкция для Nextcloud. Динамические инструкции, плейсхолдеры и тексты уведомлений меняются в зависимости от провайдера. Автоопределение провайдера при загрузке старой конфигурации без поля `provider`: если `baseUrl` содержит `yandex`, ставится `'yandex'`, иначе `'webdav'`.
*   **Обход CORS и CSP через Background Script (Service Worker):** В соответствии с политикой Manifest V3 все WebDAV-запросы контентного скрипта (`content.js`) перенаправляются через `chrome.runtime.sendMessage` в фоновый сервис-воркер `background.js`. Это полностью исключает блокировку запросов по CORS и CSP со стороны страницы `boosty.to`. В `background.js` добавлено безопасное обращение к API значков (`chrome.action || chrome.browserAction`) и в `manifest.json` добавлен блок `"action"` для корректной работы API значков. Для корректной работы юнит-тестов в среде Node.js (Vitest) сохранен фолбек на прямой сетевой `fetch`.
*   **Учётные данные:** Хранятся отдельно в `lf_webdav_config` (`baseUrl`, `username`, `accessCode`). В UI используется термин **«код доступа»**, не «пароль» — это app password/token с сервера, не пароль от аккаунта.
*   **Защита от автозаполнения и глазик для скрытия:** Поле «Код доступа» переведено в `type="text"` и скрывается с помощью CSS-свойства `-webkit-text-security: disc;`. Это предотвращает навязчивое автозаполнение и генерацию паролей браузерами. В правой части поля добавлена кнопка-глазик для переключения видимости реального кода (отслеживается через `state.ui.showAccessCode`). Кнопки очистки поля ввода (крестики) расположены бесконфликтно и сдвинуты влево.
*   **Запрет кириллицы в логине:** В поле ввода имени пользователя (`#lf-webdav-username`) установлена регулярная очистка по событию `input`, мгновенно удаляющая любые кириллические символы.
*   **Авто-отсекание почтового домена Яндекса:** При создании подключения в `createWebDavProvider`, если провайдер — `'yandex'` и имя пользователя заканчивается почтовым доменом Яндекса (например, `@yandex.ru`, `@ya.ru`), доменная часть автоматически отрезается под капотом для совместимости с требованиями Яндекса к логину WebDAV. Визуально в поле ввода введенный текст e-mail остается без изменений, а корпоративные почтовые адреса на Яндексе не затрагиваются.
*   **Оранжевый прогресс-бар:** Под кнопкой «Синхронизировать» при старте синхронизации рендерится оранжевая полоса прогресса с бесконечной плавной анимацией заполнения (до 95% за 2.5 секунды). Контейнер кнопок синхронизации использует `flex-direction: column;` для вертикального выравнивания.
*   **Сброс зависания кнопки:** Сброс флага `state.ui.webdavSyncing` перенесен в блок `finally` функции `performWebDavSync` вместе с обязательным вызовом `renderSettingsContent()` (или `render()`), гарантируя возврат кнопки в активный режим и скрытие прогресс-бара при любом исходе.
*   **Модуль:** `webdav-sync.js` — клиент WebDAV, слияние каналов, merge постов/прогресса. Подключается в `manifest.json` перед `content.js`.
*   **Формат в облаке:** ZIP-архив `boosty-bookmark/boosty_bookmark_sync.zip` — та же структура, что у ручного экспорта (`{slug}/progress.json`), включая кэш постов.
*   **Слияние данных:** Слияние `user_data` осуществляется по таймстампам `updatedAt` (статус и заметки берутся из копии с более свежим `updatedAt`). Массивы `readPosts` объединяются методом Union. Массивы `playerTimestamps` объединяются по максимальному значению, посты по `id` с сохранением `isLiked: true`, настройки — по более свежему `exportDate`.
*   **Автосинхронизация при открытии:** Вызывается `triggerAutoWebDavSync()` при открытии сайдбара, ограниченная лимитом не чаще 1 раза в минуту (`WEBDAV_AUTO_SYNC_MIN_INTERVAL_MS = 60000`). Кнопка ручной синхронизации работает без лимитов.
*   **Авто-выгрузка (debounce):** После каждого успешного `saveStateToStorage()` запускается 20-секундный таймер `debouncedWebDavUpload()`. По истечении выполняется тихая синхронизация. Повторные сохранения сбрасывают таймер.
*   **Обработка ошибок авторизации:** В методах `download` и `upload` при статус-кодах `401`/`403` выбрасывается понятная ошибка `'Неверное имя пользователя или код доступа'`.
*   **Manifest:** Добавлен сервис-воркер `background.js` в секцию `"background"`. Сохранен `host_permissions: ["https://*/*", "http://*/*"]` для доступа к серверам.
*   **UI:** Раскрывающийся блок «Синхронизация и бэкап» в меню настроек сделан сворачиваемым (collapsible) и свернут по умолчанию.
*   **Тесты:** `tests/webdav-sync.test.js` — юнит-тесты слияния (Vitest), расширенные для проверки нового слияния с `updatedAt`.

### Категория «Объявления»
*   **Исключение из общих правил:** Тайтл «Объявления» (ранее «Разное») не привязывается к обычным уровням подписки (например, «Любителям ютуба») и выводится в самом конце списка вкладки «Все» (сразу после «Бесплатные»).
*   **Особый рендеринг без детального вида:** Для него не создается обычная строка тайтла. Вместо этого в списке отображается выпадающая группа «Объявления», внутри которой сразу выводятся посты. Детальный вид, блокнот и выбор статуса отслеживания для этой категории отсутствуют.
*   **Счетчик прогресса:** В заголовке группы выводится прогресс прочитанных постов в формате `прочитано/всего` (например, `1/39`), который интерактивно обновляется при изменении чекбоксов глав.
*   **Сортировка постов:** Посты внутри «Объявлений» по умолчанию выводятся новые вверху и полностью подчиняются глобальному селектору сортировки тайтлов (по алфавиту, по дате или по прогрессу).

### Двусторонняя синхронизация лайков
*   **Из Boosty в расширение:** Внедренный в *main world* скрипт `page_script.js` перехватывает нативные запросы сайта к API реакций (`/reaction`) и по `postMessage` передает события в расширение.
*   **Из расширения в Boosty:** При отметке прочитанной главы расширение отправляет API-запрос (метод `POST` для добавления реакции, `DELETE` для удаления) на Boosty. Также посылается сообщение в `page_script.js` для программного клика по кнопке реакции на странице, чтобы синхронизировать иконку сердечка.

### SPA-навигация и управление тегами/главами в ленте
*   **Настройка открытия тегов:** Добавлена настройка «Открывать теги в текущей вкладке» (по умолчанию `true`). Если отключить, ссылки тегов открываются в новой вкладке (`target="_blank"`).
    *   **Перехват на самом сайте Boosty:** Если данная настройка включена, расширение перехватывает клики по ссылкам тегов тайтлов и в блоке «Об авторе» на оригинальной странице Boosty, убирает у них атрибут `target="_blank"` и бесшовно выполняет переход в текущей вкладке через SPA без перезагрузки страницы.
*   **Искать главы в ленте тайтла (Бета):** Добавлена настройка `openChaptersInFeed` (по умолчанию выключена, `false`). При клике на главу по умолчанию открывается прямая страница поста. При включении чекбокса под ним всплывает предупреждение о бета-статусе функции с крестиком закрытия. Включение перенаправляет пользователя на страницу тега тайтла с хэшем `#post-[id]`.
*   **SPA-навигация:** При переходе по тегу тайтла или при переходе к главе внутри одной вкладки расширение перехватывает клик и выполняет бесшовный SPA-переход без перезагрузки всей страницы Boosty (через `history.pushState` и генерацию события `popstate` для уведомления React Router сайта). SPA-переход активен для всех видов открытия глав (как прямых страниц постов, так и поиска в ленте) в текущей вкладке.
*   **Автоскролл к главе (Улучшенный):** При переходе по ссылке главы с хэшем `#post-[id]` запускается интервал до 30 попыток (каждые 500мс), который ищет пост на странице.
    * Поиск поста выполняется по регистронезависимому универсальному селектору `a[href*="${postId}" i]`, что делает его невосприимчивым к регистру UUID в URL.
    * Скроллинг страницы вниз (`window.scrollTo(0, document.documentElement.scrollHeight)`) для триггера бесконечной ленты откладывается и происходит только после 5 неудачных попыток (~2.5 секунды). Это даёт время на первичный рендеринг и исключает прыжки страницы на коротких страницах или страницах с одним постом.
*   **Сохранение состояния (sessionStorage):** Чтобы перезагрузки страницы Boosty (или SPA-переходы) не приводили к закрытию карточки детального вида тайтла или сбросу открытой вкладки расширения, текущее UI-состояние (`state.ui.activeTitle` and `state.ui.activeTab`) автоматически сохраняется в `sessionStorage` браузера для этой вкладки и восстанавливается при инициализации.

### Сброс данных расширения
*   **Двухэтапное подтверждение:** В самый низ настроек (под блоком информации о версии) добавлена скромная, невзрачная кнопка сброса данных. Для обхода блокировок браузера на вызовы системных окон `confirm()` используется встроенное двухэтапное подтверждение: при первом клике кнопка меняет статус и текст на *«Вы точно уверены? Нажмите для удаления»* (и окрашивается в красный), а также рядом появляется кнопка *«Отмена»*. Если повторный клик не совершен за 5 секунд, состояние автоматически сбрасывается.
*   **Действие сброса:** Полностью очищает локальное хранилище `chrome.storage.local.clear()`, обнуляет состояние `state` (включая масштаб), сбрасывает настройки DevTools (`devSettings`) к заводским значениям с удалением всех визуальных дев-эффектов, и направляет пользователя на приветственный экран для первой синхронизации.

### Медиаплееры (Автозапоминание времени и принудительное качество видео)
*   **Поддержка плееров:** Расширение отслеживает встроенные плееры Boosty: аудио-теги `<audio>` и видео веб-компоненты `<vk-video-player>`. Сторонние iframe не поддерживаются во избежание CORS-ошибок.
*   **Автозапоминание времени:** При обнаружении плеера расширение проверяет наличие сохраненного прогресса по уникальному идентификатору (ID видео или URL аудио). Во время воспроизведения прогресс сохраняется в базу каждые 10 секунд (троттлинг).
*   **Фильтрация и сброс:** Игнорируются медиа короче 60 секунд. Прогресс не сохраняется для первых 10 секунд и последних 10 секунд контента во время просмотра. При этом, как только воспроизведение видео или аудио доходит до самого конца (событие `ended`), сохраненная точка прогресса полностью удаляется из памяти расширения (`state.playerTimestamps`), а строка прогресса мгновенно убирается из интерфейса сайдбара, позволяя при повторном открытии начать просмотр с начала (`0:00`).
*   **Принудительная установка качества видео (Force Video Quality):** Добавлена возможность автоматически форсировать качество воспроизведения в видеоплеере VK (в текущей версии опция временно заморожена в коде и скрыта в UI до интеграции обновленной логики работы с новым плеером VK).
    *   В настройках добавлены чекбокс «Принудительное качество видео» (`forceVideoQuality`, по умолчанию `false`) и выпадающий список выбора предпочитаемого разрешения (`videoQuality`, по умолчанию `'1080p'`).
    *   Если чекбокс выключен, выпадающий список блокируется (`disabled`) и визуально приглушается (`opacity: 0.5`, `pointer-events: none`).
    *   Под капотом: при первом клике пользователя по интерактивной области видеоплеера (`div.player-wrapper div.container`) запускается интервальный опрос (каждые 100 мс в течение 5 секунд). Как только список разрешений `li.item-quality` появляется в Shadow DOM плеера, расширение безусловно кликает по соответствующему качеству (без проверки на активность, чтобы принудительно вывести плеер из режима «Авто»), после чего интервал очищается. Вся цепочка подробно логируется в консоли для упрощения отладки.
    *   Если элементы управления плеера еще не появились в Shadow DOM на момент инициализации, используется `MutationObserver` для отложенного подключения слушателя клика.
*   **Управление (Защита от конфликтов):** Опция по умолчанию включена, но в настройках добавлена галочка «Запоминать время видео и аудио». Её отключение позволяет избежать потенциальных конфликтов со сторонними расширениями (например, `More Boosty`), если пользователь решит использовать их параллельно.

### Панель разработчика (DevTools) и скрипт сборки
*   **Изоляция дебаг-кода:** Весь код, относящийся к инструментам разработчика, изолирован с помощью специальных комментариев-маркеров `/* DEV_ONLY_START */` и `/* DEV_ONLY_END */` в файлах `content.js` и `styles.css`.
*   **DevTools Панель:** 
    *   Создается кнопка `🛠️` на левой стороне экрана (`left: 0`, противоположно основному сайдбару), которая открывает левую дев-панель.
    *   Панель содержит статистику (число постов, дата последнего поста), настройки эмуляции даты отсечки публикаций (чекбокс включения и поле ввода даты), опцию быстрого скрытия блока описания профиля автора («Об авторе») для разгрузки страницы и кнопку обрезания локальной базы постов новее указанной даты.
    *   Панель использует темную тему (`lf-dark`) и полностью скрывается при уходе с целевых страниц блога.
*   **Эмуляция даты отсечки:** При включенной эмуляции и указанной дате посты с датой публикации позже отсечки отфильтровываются в процессах инкрементальной (`performIncrementalSync`), полной (`performFullSync`) и фоновой (`backgroundSync`) синхронизации.
*   **Скрытие блока «Об авторе»**: При активации дев-опции на `body` вешается класс `lf-dev-hide-about-author`, скрывающий оригинальный CSS-модуль `[class*="AboutAuthor-scss--module_content_"]` через `display: none !important`.
*   **Локальное хранение:** Настройки DevTools (`lf_dev_settings`) сохраняются локально в `chrome.storage.local` под отдельным ключом, не синхронизируются в WebDAV и не включаются в ZIP-бэкапы.
*   **Автоматизированный скрипт сборки (`build.js`):**
    *   Команда `npm run build` запускает Node.js скрипт сборки.
    *   Скрипт создает временную директорию `.tmp_build`, копирует туда файлы расширения, вырезает из всех `.js` и `.css` файлов блоки кода, находящиеся внутри маркеров `DEV_ONLY`, упаковывает результат в максимальном сжатии в архив `boosty-bookmark-release.zip` и удаляет временные файлы.
    *   Скрипт совместим с `archiver` версии `8.0.0` (использует класс `ZipArchive` напрямую).

### Локализация интерфейса
*   **Словарь locales.js:** Все текстовые надписи, заголовки, описания настроек, категории подписок и уведомления вынесены в отдельный файл `locales.js`, который внедряется как контентный скрипт перед `content.js` в `manifest.json`.
*   **Динамический выбор языка:** Язык интерфейса определяется функцией `getCurrentLang()`. Она считывает значение куки `locale` (`ru` или `en`) из `document.cookie` (доступно сразу с самого начала парсинга DOM). Если кука отсутствует, используется фолбек на `document.documentElement.lang`. Это предотвращает состояние гонки (race condition) при обновлении страницы по F5, когда React-код Boosty ещё не успел обновить атрибут `lang` у `<html>`.
*   **Функция перевода t():** Функция `t(key, ...args)` извлекает перевод по ключу и поддерживает форматирование строк с плейсхолдерами вида `{0}`, `{1}` и т.д.
*   **Локализация категорий:** Функция `tCategory(catName)` переводит названия тиров подписок на основе соответствия русским названиям, возвращая исходное название, если ключ перевода отсутствует.

## 6. Автоматизированное тестирование

### Инфраструктура
*   **Среда тестирования:** Vitest (юнит-тесты) и Playwright (E2E-тесты), окружение `jsdom` для Vitest.
*   **Конфигурация:**
    *   В `vitest.config.js` добавлено правило `include: ['**/*.test.js']`, чтобы Vitest запускал только файлы с маской `*.test.js` и игнорировал E2E-тесты Playwright (`*.spec.js`).
    *   Папка результатов тестов `test-results/` внесена в `.gitignore`, чтобы не засорять историю Git.
*   **Скрипты запуска (`package.json`):**
    *   `npm run test:unit` — запуск юнит-тестов (Vitest).
    *   `npm run test:e2e` — запуск E2E-тестов (Playwright).
    *   `npm test` — запуск обоих тестовых наборов последовательно.

### Юнит-тестирование
*   **Общие юнит-тесты (`tests/unit.test.js`):**
    *   **Моки:** Глобальные объекты расширений Chrome (`global.chrome`) и `sessionStorage` полностью мокируются перед загрузкой скрипта расширения `content.js` в CommonJS-формате.
    *   **Тесты:** Покрывают логику форматирования дат `formatDate`, сравнение постов `arePostsEqual`, создание профилей пользователей `ensureUserData` и группировку постов по тайтлам `getGroupedTitles` (включая фильтрацию технических тегов и обработку категории «Объявления»).
*   **Тесты слияния WebDAV (`tests/webdav-sync.test.js`):**
    *   **Тесты:** Покрывают логику слияния локальных и удалённых состояний по таймстампам `updatedAt` (сравнение таймстампов для заметок/статуса, объединение прочитанных постов методом Union, объединение таймстампов медиаплееров по максимальному значению).

### E2E-тестирование (`tests/extension.spec.js`)
*   **Запуск:** Тестирование в Playwright выполняется во временном браузерном контексте Chrome с флагами загрузки расширения.
*   **Мокирование сети:** Playwright перехватывает запросы к самой HTML-странице Boosty (возвращая локальный макет для автономности тестов) и API-запросам к `api.boosty.to`, исключая реальную авторизацию и интернет-зависимость.
*   **Тесты (всего 7 сценариев):**
    1.  *Инициализация:* Появление кнопки-триггера при загрузке страницы, открытие сайдбара и скрытие триггера.
    2.  *Синхронизация и отображение:* Запуск синхронизации с пустого экрана, ожидание загрузки, переход во вкладку «Все», раскрытие группы «Все» кликом по заголовку и проверка активности чекбоксов глав на основе `isLiked`.
    3.  *Поиск и фильтрация:* Поиск тайтла по подстроке во вкладке «Все» (с авто-раскрытием группы при поиске) и очистка поля кнопкой-крестиком.
    4.  *Блокнот (заметки):** Ввод текста в `#lf-notes-textarea`, автосохранение по событию `blur`, переход назад и проверка сохранения текста при повторном открытии.
    5.  *Смена статусов:* Изменение статуса тайтла на «Смотрю» в детальном виде и проверка его корректного отображения на вкладке «Смотрю».
    6.  *Очистка новинок с двухэтапным подтверждением:* Проверка открытия панели DevTools, включения эмуляции отсечки даты, установки даты на 10 мая, запуска полной синхронизации, эмуляции поступления новых постов переключением на 1 июня, проверки появления новинок на вкладке «Новые», а затем очистки с двухэтапным подтверждением.
    7.  *Сброс данных с двухэтапным подтверждением:* Проверка первого клика по кнопке удаления данных (появление подтверждения и кнопки «Отмена»), проверка отмены, и проверка полного сброса (очистка хранилища, сброс дев-настроек и возврат на стартовый экран синхронизации).
